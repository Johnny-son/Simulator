# 新一代 CPU 模拟器验证接口框架设计说明

本文档详细解释新模拟器项目的目标、技术背景、LLM 在其中的角色，以及已确定的框架设计中每个核心概念的含义。文档面向项目参与者（包括人类开发者和 LLM Agent），不假设读者已了解 gem5 或启蒙3号的内部实现。

***

## 一、项目目标与背景

### 1.1 我们要解决什么问题

传统的 CPU 架构设计依赖人类工程师手工完成——设计微架构方案、编写 RTL 代码、搭建验证环境、跑仿真收集性能数据、再根据数据调整设计。这个过程极其耗时，一个高性能乱序 CPU 核的开发周期通常以年计。

近年来出现的\*\*设计空间探索（DSE）\*\*工具试图自动化其中的"调参"环节：给定一个固定的 CPU 模板（比如一个已经实现好的乱序处理器），自动搜索最优的参数组合（ROB 大小、缓存容量、发射宽度等）。但 DSE 的局限在于——它只能在人类预先定义好的模板内搜索参数，不能发明新的微架构方案。比如，DSE 可以帮你找到最优的缓存大小，但不能帮你设计一种全新的缓存预取算法。

我们的项目要做的，是**超越传统 DSE 的微架构自动发现框架**。核心思路是：让 LLM（大语言模型，如 Claude Code）作为"架构师"和"工程师"，直接在模拟器代码中实验新的微架构设计、生成对应的硬件描述代码（RTL）、并自动验证生成结果的正确性。

### 1.2 为什么需要一个新的模拟器

目前有两个现成的模拟器可以参考，但都不能直接满足需求：

**启蒙3号模拟器**（当前仓库）是一个周期精确的 RISC-V 乱序处理器模拟器。它的优点是：

- **周期精确**：每个仿真周期精确模拟硬件的行为，能给出准确的性能数据（IPC、缓存命中率等）。
- **功能完整**：实现了完整的七级乱序流水线（取指、解码、重命名、分派、发射、执行、提交），支持 RV32IMASU 指令集（32 位），能启动 Linux。
- **硬件语义清晰**：代码严格区分组合逻辑（comb）和时序逻辑（seq），与真实硬件的行为一一对应。

但它的缺点也很明显：

- **编程复杂度高**：每个模块都要手动拆分为 comb() 和 seq() 两个函数，开发和维护负担大。对 LLM 来说，理解和修改这种风格的代码也更困难。
- **模块耦合紧密**：模块间通过共享的 IO 结构体直接通信，缺少明确的接口边界，难以独立替换或测试单个模块。
- **绑定 RISC-V**：所有数据结构都包含 RISC-V 特有的字段（如 func3、func7），无法支持其他指令集。

**gem5 模拟器**是学术界最广泛使用的处理器模拟器。它的优点是：

- **编程风格高层**：模块用面向对象的方式组织，每个模块只需实现一个 tick() 方法，代码可读性强。
- **ISA 无关设计**：后端流水线完全不知道前端用的是什么指令集，通过抽象的指令接口（StaticInst/DynInst）隔离。
- **模块化和可扩展**：通过 SimObject 基类和 Port 系统，模块可以灵活组装和替换。

但 gem5 也有不适合我们的地方：

- **不是严格的周期精确**：gem5 的 O3CPU 模型虽然建模了乱序流水线，但它的事件驱动仿真并不保证每个信号在每个周期的精确行为，无法直接用于与 RTL 做信号级对比验证。
- **无法提供模块级验证反馈**：gem5 没有设计成可以将某个模块替换为 RTL 实现并做联合仿真。

**我们的新模拟器**定位在两者之间：

- 保持**周期精确**（像启蒙3号），能给出精确的性能数据，能与 RTL 做对比验证。
- 采用**高层编程风格**（像 gem5），每个模块一个 tick() 方法，代码易读易改。
- 实现**ISA 无关后端**（像 gem5），通过通用 uOP 作为后端调度粒度，支持 x86-64 和 RV64。
- 新增**模块级验证接口**（gem5 没有的），每个模块都可以独立运行、被替换、被录制、被 Verilator 桥接。

### 1.3 通用 uOP 的意义

传统 CPU 的后端直接处理 ISA 定义的指令（如 RISC-V 的 ADD、x86 的 MOV）。不同 ISA 的指令格式、寄存器数量、寻址方式差异巨大，导致后端代码与 ISA 紧密耦合。

我们的方案是：在前端译码器中将不同 ISA 的指令翻译为**统一的通用微操作（uOP）**，后端只看到 uOP，完全不知道原始指令来自哪个 ISA。这类似于 x86 处理器内部将 x86 指令翻译为内部微操作的做法。

通用 uOP 是一组 ISA 无关的基本操作，覆盖：

- 算术/逻辑运算（加、减、与、或、移位等）
- 存储访问（加载、存储、原子操作）
- 分支/跳转
- 系统操作（CSR/MSR 读写、特权级切换等）
- 地址计算（x86 复杂寻址模式分解出的地址计算操作）

AGON 论文中的 nOP（nano-operator）是一个类似的概念——用一组基本操作符来描述指令功能，使功能描述与硬件实现解耦。我们的通用 uOP 在精神上与 nOP 类似，但需要更丰富（支持 x86 的复杂指令）、更适合周期精确建模。

> **注意**：通用 uOP 的具体设计尚未确定，是项目其他成员的工作。验证接口框架的设计不依赖 uOP 的具体字段定义，而是对其保持参数化。

> **补充（spec change-20260408-013040）**：为加快验证接口落地，本框架当前临时选用 `code/uOP/uop指令说明.md` 描述的 uOP 字典（LA64 风格）作为后端调度粒度的**占位 uOP 集合**，最终通用 uOP 集合以后端正式版本为准。占位字典的命名空间会出现在 `varification-design.md` 的 `UopCommitInfo.uop_type_placeholder`、`RflagsMask` 初始化、附录 A/B 中。后端正式 uOP 替换时，仅需替换占位枚举与初始化常量，不影响 SimModule/Port/TimeBuffer/Difftest/ModuleBridge 等核心结构。

## 二、LLM 在自动设计流程中的角色

LLM（具体来说是 Claude Code 这样的编程 Agent）在这个平台中扮演三个角色，每个角色对模拟器有不同的接口需求。

### 2.1 场景一：架构探索 Gym

**LLM 做什么**：LLM 充当"微架构师"，通过修改模拟器代码来实验新的微架构方案。

**具体流程**：

1. LLM 收到一个设计任务，例如"为这个 CPU 设计一种新的缓存预取算法"。
2. LLM 阅读模拟器中缓存模块的代码，理解当前的接口和行为。
3. LLM 编写一个新的缓存预取模块，实现自己设计的算法。
4. 新模块替换旧模块后，模拟器运行基准测试程序。
5. 模拟器返回性能指标（IPC、缓存命中率、分支预测准确率等）。
6. LLM 根据反馈调整设计，迭代优化。

**对模拟器的要求**：

- **模块可替换**：LLM 写的新模块必须能无缝替换旧模块，而不需要修改其他部分的代码。这要求每个模块有明确的接口定义。
- **代码语义明确**：LLM 需要能通过阅读代码理解模块的行为。高层的 tick() 风格比底层的 comb/seq 分离风格更容易理解。
- **独立可运行**：LLM 可能只关心一个子模块（如 TAGE 分支预测器），需要能只测试这个子模块而不运行整个 CPU。

### 2.2 场景二：RTL 生成参考

**LLM 做什么**：LLM 充当"RTL 工程师"，参考模拟器中某个模块的 C++ 实现，生成对应的 Verilog/SystemVerilog 硬件描述代码。

**具体流程**：

1. LLM 收到任务"根据模拟器中 Rename 模块的实现，生成对应的 RTL"。
2. LLM 阅读 Rename 模块的 C++ 代码，理解其功能逻辑。
3. LLM 阅读 Rename 模块的输入端口和输出端口定义，理解接口规格。
4. LLM 生成实现相同功能的 Verilog 代码，其模块接口与模拟器端口定义对应。

**对模拟器的要求**：

- **端口定义清晰**：每个模块的输入和输出必须有明确的类型定义和语义说明，LLM 才能据此生成 RTL 的端口列表。
- **功能逻辑集中**：一个模块的全部行为最好集中在一个 tick() 方法中，而不是分散在多个回调或事件处理器中，方便 LLM 一次性阅读理解。
- **ISA 无关**：后端模块的代码中不应出现 ISA 特有的概念，这样 LLM 生成的 RTL 也是 ISA 无关的。

### 2.3 场景三：RTL 验证

**LLM 做什么**：LLM 生成了 RTL 代码后，需要验证它是否正确。验证方法是：用模拟器的 C++ 模块作为参考（golden reference），与 RTL 模块做联合仿真对比。

**具体流程**：

1. LLM 生成的 Verilog 代码通过 Verilator 工具编译为 C++ 模型（Verilator 是一个开源工具，能将 Verilog 转换为可执行的 C++ 仿真模型）。
2. 在全系统仿真中，模拟器的某个 C++ 模块和对应的 RTL 模型同时运行。
3. 每个周期，两者接收相同的输入，产出各自的输出。
4. 比较两者的输出是否一致。若一致，说明 RTL 功能正确；若不一致，报告差异。

**对模拟器的要求**：

- **端口可桥接**：模拟器端口的数据格式需要能映射到 Verilator 的信号级接口（位宽、信号名）。
- **输入可注入**：需要能将相同的输入同时灌入 C++ 模块和 RTL 模块。
- **输出可采集**：需要能每周期采集两侧的输出并比较。
- **丰富的反馈**：比较结果不应只是"通过/失败"，而应指出具体哪个信号、哪个周期不一致，帮助 LLM 定位并修复 RTL 中的 bug。

## 三、验证接口的设计需求

"验证接口"不是传统意义的单元测试框架，而是一套**模块级可观测、可替换、可独立运行的接口协议**。

### 3.1 层次化分解

CPU 模拟器中的模块按层次组织：

```
Level 0: CPU 顶层
├── Level 1: Frontend（前端：取指 + 译码）
│   ├── Level 2: BPU（分支预测单元）
│   │   └── Level 3: TAGE / BTB / RAS / LoopPredictor ...
│   ├── Level 2: ICache（指令缓存）
│   ├── Level 2: Decoder（译码器，ISA 相关的唯一位置）
│   └── Level 2: FTQ（取指目标队列）
├── Level 1: Backend（后端：ISA 无关，以通用 uOP 为粒度）
│   ├── Level 2: Rename（重命名）
│   ├── Level 2: Dispatch（分派）
│   ├── Level 2: IssueQueue（发射队列，可能有多个）
│   ├── Level 2: ExecUnit（执行单元，可能有多个）
│   │   └── Level 3: ALU / MulDiv / BranchUnit / ...
│   ├── Level 2: ROB（重排序缓冲区）
│   ├── Level 2: PRF（物理寄存器文件）
│   └── Level 2: LSU（访存单元）
├── Level 1: MemorySubsystem（存储子系统）
│   ├── Level 2: DCache（数据缓存）
│   ├── Level 2: PTW（页表遍历器）
│   └── Level 2: LLC（末级缓存，可选）
└── Level 1: 验证/反馈层（横切所有模块）
```

层次化的意义在于：LLM 可以在任意粒度上操作。如果任务是"设计一种新的分支预测算法"，LLM 只需要关注 Level 3 的 TAGE 模块；如果任务是"优化整个前端的取指效率"，LLM 关注 Level 1 的 Frontend。每一层的模块都应该能独立运行和验证。

### 3.2 模块可替换性

每个模块通过统一的接口规范定义其输入和输出。替换一个模块时，只要新模块实现了相同的接口，就可以直接嵌入系统运行，不影响其他模块。

这是通过 C++ 的多态机制实现的：所有模块继承同一个基类（类似 gem5 的 SimObject），基类定义了标准的方法签名（如 tick()、getPort() 等）。替换模块就是提供一个新的子类实现。

### 3.3 模块可独立运行

"独立运行"意味着可以只启动一个模块进行测试，而不需要搭建整个 CPU。这有两种实现方式：

**方式一：存根（Stub）驱动**

被测模块的上下游模块用存根替代。存根是一种简化实现——它只提供最基本的输入（如每周期固定发送 N 条指令）或无条件接受输出。这样被测模块可以在一个简化环境中运行。

**方式二：Trace 回放驱动**

先在全系统仿真中录制被测模块的输入/输出序列（trace），然后用录制的输入序列驱动被测模块独立运行，检查其输出是否与录制的输出一致。

两种方式互补：存根驱动适合探索性测试（试验新设计），trace 回放适合回归验证（确认修改没有破坏已有行为）。

**方式三：定向 I/O 测试**

设计者为模块编写针对性的测试用例：构造特定的输入序列驱动模块独立运行，检查输出是否符合预期。与存根驱动的区别在于，定向 I/O 测试的输入不是"最基本的固定输入"，而是针对特定功能场景精心设计的激励。例如：

- 对分支预测器：构造特定的分支模式（全取、全不取、交替取），检查预测结果是否符合算法预期。
- 对 ROB：构造恰好填满 ROB 的指令序列，检查满队列后是否正确阻塞上游。
- 对乘法器：构造边界值输入（如最大值溢出），检查结果和输出时序（是否恰好在预期的周期输出）。

#### 模块可测试性设计原则

上述三种独立运行方式对模块设计提出一个统一要求：**每个模块应设计成容易独立调用、容易编写定向 I/O 测试和运行时断言的形式**。这不是可选的"最佳实践"，而是服务于模拟器正确性验证策略的硬性设计约束（详见 4.3）。具体地：

- **接口清晰**：模块的输入和输出通过明确的端口定义，不依赖全局状态或隐式的共享变量。模块通过端口与外界通信，而非直接访问其他模块的内部数据。
- **可独立实例化**：模块可以脱离完整 CPU 环境单独创建，上下游用存根替代即可运行。不需要初始化整个 CPU 才能测试一个子模块。
- **状态可观测**：模块的关键内部状态（如队列深度、计数器值、有限状态机当前状态）可以被测试代码查询，用于编写断言和检查。
- **确定性**：相同的输入序列产生相同的输出序列，不依赖未初始化的状态或外部随机源。

### 3.4 Verilator 联合仿真桥接

当需要验证 LLM 生成的 RTL 时，需要一个"桥接层"（Bridge）将模拟器的 C++ 接口与 Verilator 编译出的 C++ 模型对接。

桥接的工作原理：

```
           C++ 模拟器模块            Verilator RTL 模型
                │                         │
    输入端口 ──→ [Bridge 层] ──→ RTL 输入信号
                │                         │
          tick() 执行           eval() 执行
                │                         │
    输出端口 ←── [Bridge 层] ←── RTL 输出信号
                │                         │
          比较两侧输出，生成差异报告
```

Bridge 层负责：

1. 将模拟器端口的逻辑数据类型（如"一条 uOP 指令结构体"）转换为 Verilator 的信号级数据（如"一组位宽确定的信号线"）。
2. 确保 C++ 的 tick() 调用和 Verilator 的 eval() 调用在同一个逻辑时钟周期。
3. 每周期对比两侧输出，记录差异。

### 3.5 与启蒙3号现有 difftest 的区别

启蒙3号已有一个差分测试框架（difftest），它的工作方式是：乱序流水线每提交一条指令，就与一个简单的参考模型（RefCpu）比较。这是**指令提交级别**的验证——只关心最终结果是否正确，不关心中间过程。

我们的验证接口是**模块级别**的验证——关心每个模块在每个周期的输入和输出是否正确。这是因为我们的目标不仅是验证最终结果，更是验证 LLM 生成的 RTL 在微架构级别与 C++ 参考实现行为一致。

## 四、正确性验证方法论

### 4.1 核心认知：模拟器是规格而非实现

在讨论验证方法之前，必须先确立一个关键认知：**在本项目中，模拟器是 RTL 的规格说明（specification），而非某个硬件的模拟实现。**

传统处理器开发中，验证的方向是"RTL 是否正确实现了规格"，规格由架构手册和微架构文档定义。但在我们的项目中，因果关系是反过来的：

- **BSD 生成 RTL 时**：BSD 算法直接从模拟器模块的 I/O 真值表推导布尔电路。模拟器定义了"(输入 + 当前状态) → (输出 + 下一状态)"的映射，BSD 生成的电路必须在每组输入上与模拟器的输出 bit 级一致。
- **LLM 生成 RTL 时**：LLM 阅读模拟器模块的代码和端口定义来生成 Verilog。模拟器是 LLM 的唯一参考规格。

因此：**RTL 的"正确"被定义为"与模拟器模块的行为一致"。不存在外部标准来评判模拟器自身的周期行为是否"精确"——模拟器定义了什么是"正确的周期行为"。**

这个认知简化了验证策略。我们不需要证明"模拟器在每个周期都与某个外部参考一致"，而需要证明三件事：

1. 模拟器**功能正确**——能正确执行程序（通过 difftest 验证）。
2. 模拟器**内部一致**——各模块之间的时序配合正确（通过断言和构造性测试验证）。
3. 生成的 **RTL 与模拟器一致**——模块级联合仿真验证。

但这不意味着周期级行为可以随意。模拟器的时序行为最终会被转化为 RTL，如果时序建模有 bug（如某个模块延迟错误），生成的 RTL 会继承这个 bug。验证的重点是：**模拟器的时序行为是否符合设计者的意图**。

#### 模拟器的建模层次：高层逻辑 + 时序计数

模拟器实现周期精确的方式是**高层逻辑 + 时序计数**：用 C++ 高层代码完成功能计算，用周期计数模拟硬件的流水线延迟。不需要在寄存器传输级上模拟每个触发器的赋值行为。

以一个 3 拍延迟的乘法器为例：

- **模拟器**：在输入到达的周期，用一步 C++ 乘法算出结果，然后设置 3 拍计数器。计数器到期后将结果写入输出端口。
- **RTL**：真正实现 3 级流水线寄存器，操作数在中间级逐步传递，第 3 级产出结果。

两者的内部状态完全不同（计数器 vs 流水线寄存器），但对外的 I/O 时序一致——都在输入后第 3 个周期产出有效输出。

这个建模层次对验证体系产生两个根本性约束：

1. **模块级验证只能在 I/O 边界进行**。模拟器和 RTL 的内部实现不可比较——一个是高层抽象，一个是寄存器级实现。"周期精确"在本项目中的含义是**模块的输入输出时序精确**，而非内部每个信号都与 RTL 一一对应。
2. **模拟器自身正确性不能依赖内部状态的形式化比较**，而依赖两个支柱（详见 4.3）：整体 difftest 验证功能正确性 + 人工编写的模块级检查验证各模块的行为正确性。

### 4.2 Difftest：模拟器作为 DUT

Difftest（差分测试）验证模拟器在指令语义层面的正确性：与参考模型逐指令比较提交结果。

#### 双 ISA 带来的核心挑战

现有 difftest 实现（启蒙3号、香山/XS-GEM5）都绑定单一 ISA。我们的模拟器需要同时支持 x86-64 和 RV64，这引入了三个新挑战：

1. **状态空间不同**：RV64 有 32 个 64 位 GPR + 32 个 64 位 FPR + CSR + 32 个向量寄存器；x86-64 有 16 个 64 位 GPR + RFLAGS + 16 个 128 位 XMM + 段寄存器 + MSR。两者的寄存器数量、含义、编码方式完全不同。
2. **参考模型不同**：RISC-V 可用 Spike/NEMU（XS-GEM5 已验证）；x86-64 没有现成的轻量级参考模型可直接集成。
3. **指令边界不同**：RISC-V 指令固定 4 字节（或压缩 2 字节）；x86-64 指令变长 1-15 字节。uOP 拆分的粒度也不同（x86-64 的复杂指令拆分更多 uOP）。

模拟器首先需要证明自己能正确执行程序。方法是将模拟器作为 DUT，与一个功能正确的参考模型做逐指令比较。

**参考模型选择**：

| ISA 模式 | 参考模型        | 集成方式                        | 成熟度                 |
| ------ | ----------- | --------------------------- | ------------------- |
| RISC-V | Spike       | 编译为共享库，dlopen 动态加载          | 高（XS-GEM5 已验证，单核优先） |
| RISC-V | NEMU        | 编译为共享库，dlopen 动态加载          | 高（XS-GEM5 已验证，支持多核） |
| RISC-V | librefcpu.a | 静态链接（启蒙3号 Task2 正在统一的参考模型库） | 中（API 正在稳定中）        |
| x86-64 | QEMU TCG    | 封装为共享库，dlopen 动态加载          | 需从零集成               |

**触发时机**：每当 ROB 提交一条完整的宏指令（即该指令的所有 uOP 都已提交）时，触发一次 difftest 比较。

**新模拟器的 ISA 功能基线**（对标 XS-GEM5）：

新模拟器的功能目标不再限于启蒙3号的 RV32 范围，而是对标 XS-GEM5 的能力水平，同时支持 RV64 和 x86-64 两种 ISA。当前为单核设计，但架构需保证多核可扩展性。

**RISC-V 功能基线**：

| 扩展              | 说明                             |
| --------------- | ------------------------------ |
| RV64I           | 全部基础 64 位整数指令                  |
| RV64M           | 乘除法（64 位 MUL/MULH/DIV/REM 全系列） |
| RV64A           | 原子操作（LR/SC + 全部 AMO，64 位）      |
| RV64F           | 单精度浮点（独立 32 个浮点寄存器）            |
| RV64D           | 双精度浮点（独立 32 个浮点寄存器）            |
| RV64C           | 压缩指令（16 位编码）                   |
| RVV             | 向量扩展（32 个向量寄存器，宽度可配置）          |
| Zba/Zbb/Zbc/Zbs | 位操作扩展                          |
| 特权模式 M/S/U      | 三级特权，异常/中断委托                   |
| H 扩展（可选）        | 虚拟化（VS/VU 模式），按需支持             |
| SV39/SV48 虚拟内存  | 三/四级页表遍历，多级 TLB                |
| 外设              | UART、PLIC、CLINT                |

**x86-64 功能基线**（初始目标）：

| 功能       | 说明                            |
| -------- | ----------------------------- |
| 基本整数指令集  | x86-64 长模式整数指令（16 个 64 位 GPR） |
| SSE/SSE2 | 128 位 SIMD（16 个 XMM 寄存器）      |
| x87 FPU  | 80 位浮点栈（8 个栈寄存器）              |
| 保护模式/长模式 | 特权级 Ring 0-3                  |
| 4 级页表    | PML4 分页（CR3 指向 PML4 基址）       |
| 外设       | UART、PIC/APIC、PIT             |

> **参考**：启蒙3号已实现 RV32I/M/A/B（Zbb/Zba/Zbc/Zbs）、Zfinx（浮点在整数寄存器上）、M/S/U 特权模式、SV32 虚拟内存等，可作为 RV64 实现的功能参考。

以下分 ISA 说明需要对比的状态。

**RISC-V 模式对比的状态**（参考 XS-GEM5 的 `riscv64_CPU_regfile` 和 Spike/NEMU 实践）：

RV64F/D 引入独立的浮点寄存器文件（FPR），不再使用 Zfinx（浮点在整数寄存器上）的方式，因此 FPR 需要独立比较。

| 状态                        | 内容                                                                                                                       | 比较时机                                |
| ------------------------- | ------------------------------------------------------------------------------------------------------------------------ | ----------------------------------- |
| **PC**                    | 64 位程序计数器                                                                                                                | 指令执行前（检查两侧执行同一条指令）                  |
| **特权级**                   | M/S/U（可选 VS/VU）                                                                                                          | 每条指令                                |
| **GPR**                   | 32 个 64 位通用寄存器（x0 恒为 0 可跳过）                                                                                              | 每条指令                                |
| **FPR**                   | 32 个 64 位浮点寄存器                                                                                                           | 浮点指令后                               |
| **CSR（基本集）**              | mstatus、sstatus、mepc、sepc、mcause、scause、mtval、stval、mtvec、stvec、satp、mip、mie、mscratch、sscratch、mideleg、medeleg           | 每条指令                                |
| **CSR（Hypervisor 扩展，可选）** | mtval2、mtinst、hstatus、hideleg、hedeleg、hcounteren、htval、htinst、hgatp、vsstatus、vstvec、vsepc、vscause、vstval、vsatp、vsscratch | 虚拟化模式下                              |
| **向量寄存器**                 | 32 个向量寄存器（每个 VLEN 位宽，由 vlenb 决定）                                                                                         | 向量指令后                               |
| **向量 CSR**                | vtype、vl、vcsr（含 vxsat、vxrm）、vlenb                                                                                        | 向量指令后                               |
| **存储写入**                  | 地址（物理地址）、数据、写掩码                                                                                                          | 存储指令提交时                             |
| **LR/SC 保留状态**            | reserve\_valid、reserve\_addr                                                                                             | 多核下单独同步（通过 `SyncState` 结构），单核下通常不需要 |

不比较：mcycle/minstret（取决于微架构，通过 `skipPerfCntCsr()` 跳过）、time/timeh（取决于物理时间）、vstart（XS-GEM5 当前不比较）、内存全量（通过存储写入逐条比较覆盖，仅出错时全量对比）。

**x86-64 模式对比的状态**：

x86-64 模式需要对比的架构状态与 RISC-V 有根本差异——寄存器数量不同、标志位语义不同、特权模型不同。以下列出 x86-64 模式下需要比较的状态：

| 状态          | 内容                                                              | 比较时机                                        |
| ----------- | --------------------------------------------------------------- | ------------------------------------------- |
| **RIP**     | 64 位程序计数器                                                       | 指令执行前                                       |
| **特权级**     | Ring 0-3（当前特权级 CPL）                                             | 每条指令                                        |
| **GPR**     | 16 个 64 位通用寄存器（RAX、RCX、RDX、RBX、RSP、RBP、RSI、RDI、R8-R15）          | 每条指令                                        |
| **RFLAGS**  | 64 位状态标志（CF、ZF、SF、OF、PF、AF 等）                                   | 每条修改标志的指令（需按指令类型生成掩码，只检查被修改的位，如 INC 不修改 CF） |
| **段寄存器**    | CS、SS、DS、ES、FS、GS 的选择子                                          | 段操作指令后（远调用/远跳转/IRET 等）                      |
| **控制寄存器**   | CR0（保护模式/分页控制）、CR2（缺页线性地址）、CR3（PML4 基址）、CR4（扩展特性）、CR8（任务优先级，可选） | CR0/CR3/CR4：特权指令后；CR2：缺页异常时                 |
| **x87 FPU** | 8 个 80 位浮点栈寄存器 + x87 状态字/控制字                                    | x87 浮点指令后                                   |
| **XMM**     | 16 个 128 位 SSE 寄存器                                              | SSE 指令后                                     |
| **MXCSR**   | SSE 控制/状态寄存器（舍入模式 bits 13-14、异常掩码 bits 7-12、异常标志 bits 0-5）      | SSE 浮点指令后                                   |
| **异常状态**    | 异常号 + 错误码（#PF、#GP、#SS 等异常会压入错误码，指示读/写/取指和用户/内核）                 | 异常发生时                                       |
| **存储写入**    | 地址（物理地址）、数据、写掩码                                                 | 存储指令提交时                                     |

关于 MXCSR：功能上对应 RISC-V 的 `fcsr`（浮点控制/状态寄存器），两者都管理舍入模式和异常标志。

关于存储写入：单核 difftest 中，两个 ISA 的存储比较机制相同（addr/data/mask 在提交时逐条比较）。TSO（x86-64）与 RVWMO（RISC-V）的内存模型差异只影响多核场景下存储的全局可见顺序，不影响单核提交时的比较逻辑。

关于原子操作：x86-64 使用 LOCK CMPXCHG（单条原子指令）而非 RISC-V 的 LR/SC（两条指令 + 跨指令 reservation 状态），因此 x86-64 不需要类似 LR/SC 的 reservation 状态同步。

不比较：TSC（时间戳计数器，取决于物理时间）、性能计数器 MSR（通过 `skipPerfCntCsr()` 机制跳过）、段描述符缓存的隐藏部分（base/limit/attribute 是微架构可见但架构隐藏的状态）。

**ISA 独立的状态结构体设计**（参考 XS-GEM5 的 `riscv64_CPU_regfile` 和 NEMU 哨兵模式）：

每个 ISA 定义独立的状态结构体。RISC-V 侧参考 XS-GEM5 的设计——将 GPR（32 个 64 位）、FPR（32 个 64 位）、CSR（基本集 + Hypervisor 扩展 + 向量扩展）、向量寄存器（32 个，宽度由 VLEN 决定）组织在一个结构体中。x86-64 侧需要独立定义（16 个 64 位 GPR、RFLAGS、16 个 128 位 XMM、控制寄存器、x87 FPU 状态等）。

对于 uOP 架构，存储比较的单位是宏指令而非 uOP——一条存储宏指令的所有 uOP 都提交后才触发比较。

**Difftest API 与 Proxy 工厂**：

采用 XS-GEM5 验证的标准 difftest API，通过 Proxy 工厂模式在运行时按 ISA 加载不同的参考模型（RISC-V → Spike 或 NEMU，x86-64 → QEMU TCG）。完整的 API 包括：

- `memcpy` / `memcpy_init`：内存同步
- `regcpy`：GPR + FPR 寄存器同步
- `csrcpy`：CSR 寄存器同步
- `exec(n)`：执行 n 条指令
- `guided_exec(guide)`：引导执行（异常处理）
- `raise_intr(no)`：中断注入
- `uarchstatus_cpy`：微架构状态同步（LR/SC 的 reservation 状态）
- `update_config`：动态配置更新（如调试模式切换）
- `query`：查询参考模型内部状态
- `isa_reg_display`：调试用寄存器显示

所有 API 使用方向标志（`DIFFTEST_TO_DUT` / `DIFFTEST_TO_REF`）区分数据流向。

具体的状态结构体定义、API 签名和 Proxy 类设计见 `varification-design.md` 第二章。

#### uOP 过滤与融合指令：从 uOP 提交到指令级 difftest

后端以 uOP 为粒度运行，ROB 中存的是 uOP 而非宏指令。但 difftest 在宏指令粒度比较。XS-GEM5 已验证了一种简洁有效的处理模式：

**uOP 过滤**：每条 uOP 在流水线中携带两个标志——`isMicroop()`（是否为某条宏指令拆分出的 micro-op）和 `isLastMicroop()`（是否为该宏指令的最后一条 micro-op）。提交级在每条 uOP 提交时检查：只有当 `!isMicroop() || isLastMicroop()` 为真时才触发 REF 执行和状态比较。中间 micro-op 直接跳过，不执行 REF，不做比较。这种过滤方式比此前设想的"聚合层"更简洁——不需要维护 inst\_seq 到 uOP 的映射关系，不需要显式的聚合数据结构。

**融合指令处理**：当前端将两条相邻的原始指令融合为一条内部指令时（如 XS-GEM5 中的 `isFusion()` 标记），difftest 需要让 REF 连续执行两步以保持同步。每条融合指令在提交时调用 `exec(1)` 两次。

**冲刷处理**：当宏指令被冲刷（squash）时，该指令的所有 uOP 都会被取消，不会有"部分 uOP 已提交、部分取消"的情况——ROB 保证按序提交，冲刷发生在提交之前。因此不需要回滚 REF 状态。

> **补充（spec change-20260408-013040）**：宏指令到 uOP 的两级分解由前端 IDU + 后端 Dispatch 负责（参考 `verification-backend-api.md` §1 中的 `macro_id` + `last_uop` 字段）。本节描述的 difftest 过滤逻辑只依赖 `last_uop` 标志，**不依赖具体的 uOP 类型枚举**——这一点对当前的占位 uOP 字典或未来更换通用 uOP 字典都成立。`isMicroop()`/`isLastMicroop()` 的实现可以直接读取 DynInst 中的 `last_uop` 位。

#### DUT 与 REF 不完全同步的情形及处理方式

以下列出 difftest 中需要特殊处理的不同步场景。场景分类基于 XS-GEM5 的实际实现（已在生产环境中验证）、新模拟器的架构特点（uOP 后端、外部 .so 参考模型、RV64 + x86-64 双 ISA），以及参考模型的选择（Spike/NEMU、QEMU TCG）。

**单核必需场景**

以下表格统一列出所有需要特殊处理的场景，包括双 ISA 共有场景和 ISA 特有场景。每个场景在表格中给出要点，部分复杂场景在表格后有展开说明。

| 场景                  | 适用 ISA | 处理方式                                                                                                                                          |
| ------------------- | ------ | --------------------------------------------------------------------------------------------------------------------------------------------- |
| **uOP 过滤**          | 共有     | 通过 `isMicroop()` / `isLastMicroop()` 标志过滤：只在宏指令的最后一条 micro-op 提交时触发 REF 执行和比较，中间 micro-op 跳过。详见上节"uOP 过滤与融合指令"。                               |
| **融合指令**            | 共有     | 通过 `isFusion()` 标志识别：两条原始指令被前端融合为一条内部指令时，difftest 让 REF 连续执行两步。详见上节。                                                                          |
| **MMIO 访问**         | 共有     | 通过 `strictlyOrdered` 标记识别。跳过 REF 执行，直接用 DUT 结果同步 REF 的 PC 和目标寄存器值，再调用 `regcpy()` 写入 REF。必须覆盖全部 MMIO 地址范围。                                     |
| **中断显式注入**          | 共有     | DUT 在提交级响应中断时调用 `raise_intr(中断号)` 通知 REF，注入后将 DUT 完整状态同步到 REF。RISC-V 中断号为 mcause 值（最高位=1），x86-64 为 IDT 向量号。展开说明见下。                            |
| **异常引导执行**          | 共有     | 通过 `guided_exec()` 传递 `ExecutionGuide` 结构体（异常编号 + 附带值 + 可选跳转目标）。RISC-V 附带值为 mtval/stval；x86-64 附带值为 CR2 + 错误码。展开说明见下。                         |
| **系统指令 CSR/MSR 跳过** | 共有     | 维护 `skipCSRs` 列表。RISC-V 覆盖 ecall/mret/sret/csrrw 系列；x86-64 覆盖 MOV CR/WRMSR/LGDT/LIDT 等。匹配时跳过 CSR/MSR 比较，将 DUT 值同步到 REF。展开说明见下。                |
| **性能计数器/时间戳跳过**     | 共有     | RISC-V：跳过 mcycle、minstret、mhpmcounter3-31 等 CSR。x86-64：跳过 TSC（IA32\_TIME\_STAMP\_COUNTER）、IA32\_PERF\_\* 系列、IA32\_TSC\_AUX。匹配时将 DUT 值同步到 REF。 |
| **PC 不匹配恢复**        | 共有     | PC 不匹配但 npc 匹配时，让 REF 多执行一步再重新比较。通过则视为正常，否则报错终止。展开说明见下。                                                                                       |
| **NaN Boxing 容忍**   | RISC-V | RV64 独立 FPR 下，32 位浮点结果写入 64 位寄存器时高 32 位填充可能不同。比较时若差异仅为高 32 位且值为 `0xFFFFFFFF00000000`，视为等价并忽略。                                                 |
| **RFLAGS 部分更新掩码**   | x86-64 | x86-64 不同指令只修改 RFLAGS 的部分位，且部分位可能"未定义"。需维护"opcode 类别 → 修改标志位掩码"映射表，只检查掩码覆盖的位，未定义标志位排除在比较之外。展开说明见下。                                            |
| **段寄存器隐藏部分排除**      | x86-64 | 段寄存器包含可见的 16 位选择子和隐藏的描述符缓存（base/limit/attributes）。Difftest 只比较选择子。隐藏部分是微架构细节，DUT 和 REF 的查表时机可能不同。长模式下 FS/GS base 通过 MSR 同步。                   |
| **x87 FPU 栈状态**     | x86-64 | x87 栈由 TOP 指针（状态字 bits 11-13）映射 8 个 80 位寄存器。比较时 TOP 必须一致，空寄存器（Tag 字标记）不参与值比较。transcendental 函数（FSIN 等）可能有精度差异，需容忍最低有效位差异或跳过。展开说明见下。           |
| **MXCSR 部分更新**      | x86-64 | SSE 控制/状态寄存器中的异常标志位（bits 0-5）按指令类型选择性修改，需类似 RFLAGS 的掩码比较机制。舍入模式位（bits 13-14）和异常掩码位（bits 7-12）由显式写入指令（LDMXCSR）修改，可全量比较。展开说明见下。                 |
| **x86-64 MSR 跳过列表** | x86-64 | 除性能计数器外，以下 MSR 也需跳过：IA32\_APIC\_BASE（依赖外设配置）。CR8（任务优先级）是否比较取决于 APIC 模型一致性。展开说明见下。                                                             |

**未来扩展场景**（单核基线完成后按需添加）：

| 场景                      | 适用 ISA | 说明                                                                         |
| ----------------------- | ------ | -------------------------------------------------------------------------- |
| **多核 LR/SC 同步**         | RISC-V | 通过 `SyncState` 结构（`lrscValid` + `lrscAddr`）在 SC 执行前将 reservation 状态传递给 REF |
| **Golden Memory**       | 共有     | 多核环境下，Load/AMO 返回值可能因一致性协议与 REF 不同，使用 Golden Memory 仲裁                     |
| **L1D Cache Refill 验证** | 共有     | 多核环境下，检查缓存行 refill 数据与 Golden Memory 一致                                    |

#### 部分场景的展开说明

**PC 不匹配的产生原因与恢复机制。** Difftest 在每条指令提交时首先比较 DUT 和 REF 的当前 PC（即两侧即将执行的指令地址）。正常情况下 PC 应当完全一致，但存在几种合法的一拍偏差场景会导致 PC 暂时不匹配：（1）**系统指令的副作用时序差异**——某些系统指令（如 mret/sret、IRET）在执行后会修改 PC（跳转到 mepc/sepc 保存的地址或从栈中弹出返回地址），DUT 和 REF 对"PC 何时更新为新值"的建模可能存在一拍偏差。例如 DUT 的提交级在当前 tick() 结束后才将 PC 更新为 mepc 的值，而 REF 在执行 mret 后立即更新了 PC——此时在比较时刻，DUT 报告的 PC 仍是 mret 指令本身的地址，而 REF 已经指向了 mepc 的值。（2）**异常/中断处理的入口计算差异**——DUT 和 REF 对中断向量表地址的计算可能因 CSR 同步延迟而短暂不同。（3）**分支延迟槽**（RISC-V 无此问题，但属于通用 difftest 框架需要考虑的情况）。恢复机制的逻辑是：当发现当前 PC 不匹配时，检查 DUT 的 npc（下一条指令的 PC）是否与 REF 的当前 PC 一致。如果一致，说明偏差仅为一拍的时序差异——让 REF 多执行一步后重新比较，两侧即可重新对齐。如果 npc 也不匹配，则说明两侧的执行路径已经实质性分歧，报错终止。这种"多走一步再看"的策略避免了因时序建模差异导致的大量误报，同时不会掩盖真正的功能错误。

**REP 前缀指令的不同步含义。** x86-64 的 REP/REPE/REPNE 前缀将一条字符串操作指令（如 MOVSB、STOSB、CMPSB、SCASB）转为循环执行：每次迭代执行一次基本操作、递减 RCX（计数器）、更新 RSI/RDI（源/目标指针），直到 RCX 归零（REP）或比较条件不满足（REPE/REPNE）。问题在于 DUT 和 REF 对 REP 指令的执行粒度不同。DUT 作为微架构模拟器，会将 `REP MOVSB`（假设 RCX=100）展开为 100 条 micro-op，每条 micro-op 完成一次字节复制、一次 RCX 递减、一次 RSI/RDI 更新，这些 micro-op 在流水线中逐条执行和提交。而 REF（QEMU TCG）将整个 `REP MOVSB` 视为一条原子指令——调用 `exec(1)` 后 QEMU 一次性完成全部 100 次迭代，RCX 直接变为 0，RSI/RDI 直接指向最终位置。如果 DUT 在每条 micro-op 提交时都触发 difftest 比较，则第 1 条 micro-op 提交后 DUT 的 RCX=99，而 REF 的 RCX=0——产生严重不匹配。处理方式取决于 DUT 的微架构实现：如果 DUT 将 REP 展开为多条 micro-op（最常见的实现），则通过 `isMicroop()` / `isLastMicroop()` 标志天然处理——只有最后一条 micro-op 提交时才触发 REF 执行和比较，此时 DUT 的 RCX 也已归零，两侧状态一致。如果 DUT 将 REP 作为单条指令在执行单元中循环执行（某些简单实现），则直接在指令提交时触发一次 difftest，两侧都一次性完成，不存在不匹配问题。

**RFLAGS 部分更新掩码的具体机制。** x86-64 的标志位行为分三类：（1）定义并修改——必须比较（如 ADD 的全部算术标志）；（2）定义为特定值——必须比较（如 AND 将 CF/OF 清零）；（3）未定义——不参与比较（如 IMUL 后的 SF/ZF/PF/AF）。掩码表按指令类别组织而非逐个 opcode，因为同类指令通常影响相同的标志位集合。典型分类：

| 指令类别                | 修改的标志位                 | 未定义的标志位                |
| ------------------- | ---------------------- | ---------------------- |
| 算术（ADD/SUB/CMP/NEG） | CF, OF, SF, ZF, PF, AF | 无                      |
| 逻辑（AND/OR/XOR/TEST） | SF, ZF, PF（CF/OF 清零）   | AF                     |
| 移位（SHL/SHR/SAR）     | CF, OF, SF, ZF, PF     | AF                     |
| 自增减（INC/DEC）        | OF, SF, ZF, PF, AF     | 无（CF 不修改）              |
| 乘法（MUL/IMUL）        | CF, OF                 | SF, ZF, PF, AF         |
| 除法（DIV/IDIV）        | 无                      | CF, OF, SF, ZF, PF, AF |

**x87 FPU 栈状态比较细节。** x87 使用独特的栈模型，给 difftest 带来额外复杂度：

- TOP 指针（状态字 bits 11-13）确定 ST(0) 对应的物理寄存器编号，TOP 偏移导致整个栈映射不同——TOP 不一致意味着所有 ST(i) 值都不可比较，应视为严重错误
- Tag 字（每个寄存器 2 位）标记有效/零/特殊/空，标记为空的寄存器不参与值比较
- 80 位扩展精度（1 位符号 + 15 位指数 + 64 位尾数）是 x87 特有的，IEEE 754 没有精确的等价定义；DUT 和 REF 对 transcendental 函数（FSIN/FCOS/FPATAN/F2XM1 等）的实现精度可能不完全一致——Intel 手册只保证这些函数的误差不超过 1 ULP，不保证位精确；处理策略是对这类函数容忍最低 1-2 位尾数的差异，或在不影响正确性验证的前提下跳过这类指令的浮点寄存器比较

**MXCSR 的含义与部分更新问题。** MXCSR（MXCSR Register，全称为 Multimedia Extensions Control and Status Register）是 x86-64 架构中专门管理 SSE/SSE2 浮点运算行为的 32 位控制/状态寄存器。它在功能上对应 RISC-V 的 `fcsr`（浮点控制/状态寄存器）——两者都管理浮点舍入模式和异常标志，但 MXCSR 只控制 SSE 指令（128 位 XMM 寄存器上的 SIMD 浮点运算），不影响 x87 FPU 指令（x87 有自己独立的控制字和状态字）。MXCSR 的位布局如下：bits 0-5 为异常标志位（IE 无效操作、DE 非规格化、ZE 除零、OE 上溢、UE 下溢、PE 精度），记录最近一次 SSE 浮点运算是否触发了对应异常；bits 7-12 为异常掩码位，控制对应异常是否被屏蔽（屏蔽后异常不触发 #XM 异常，只置标志位）；bits 13-14 为舍入模式（00=最近偶数、01=向下、10=向上、11=截断）；bit 6 为 DAZ（Denormals Are Zeros，将非规格化输入视为零）；bit 15 为 FTZ（Flush to Zero，将非规格化结果刷为零）。Difftest 中 MXCSR 的部分更新问题来源于：不同 SSE 指令只修改异常标志位中与自己相关的位（例如 ADDPS 可能只置位 OE 和 PE，不影响 ZE），而舍入模式和掩码位只由显式写入指令（LDMXCSR / FXRSTOR）修改。因此比较时需要类似 RFLAGS 的掩码机制——对异常标志位按指令类型生成掩码，只检查本条指令可能修改的标志位；对舍入模式和掩码位可以全量比较。

**x86-64 MSR 的含义与跳过策略。** MSR（Model-Specific Registers，型号特定寄存器）是 x86 架构中一组数量庞大的特殊寄存器，用于控制和监视处理器的各种底层行为。与通用寄存器（RAX 等）和控制寄存器（CR0-CR4）不同，MSR 不通过普通指令访问，而是通过专用的 RDMSR（读取）和 WRMSR（写入）指令访问——指令以 ECX 寄存器中的 32 位地址编号来指定读写哪个 MSR，结果/数据通过 EDX:EAX 传递。MSR 的数量因处理器型号而异（因此得名"型号特定"），现代 Intel/AMD 处理器定义了数百个 MSR，覆盖以下几类功能：（1）**性能监控**——IA32\_PERF\_GLOBAL\_CTRL、IA32\_FIXED\_CTR0-2 等，用于配置和读取硬件性能计数器；（2）**时间戳**——IA32\_TIME\_STAMP\_COUNTER（TSC），记录处理器启动以来的时钟周期数，RDTSC 指令实际上就是在读取这个 MSR；（3）**电源管理**——IA32\_PERF\_STATUS、IA32\_PERF\_CTL，控制处理器频率和电压；（4）**系统配置**——IA32\_APIC\_BASE（配置本地 APIC 的基地址和使能状态）、IA32\_EFER（扩展特性使能，如长模式使能位 LME）；（5）**段基址**——IA32\_FS\_BASE、IA32\_GS\_BASE、IA32\_KERNEL\_GS\_BASE，在长模式下直接设置 FS/GS 段的基地址（替代通过描述符表间接设置）。在 difftest 中，MSR 的跳过策略基于一个核心原则：**依赖微架构实现细节或物理时间的 MSR 不比较**。性能计数器类 MSR 的值取决于实际执行了多少条微操作、经历了多少次缓存缺失等微架构事件，DUT 和 REF 的微架构完全不同，这些值不可能一致。TSC 取决于物理时钟，同理不可比较。IA32\_APIC\_BASE 的值取决于外设模型的配置，DUT 和 REF 的外设初始化可能不同。对于这些被跳过的 MSR，处理方式与 CSR 跳过相同——不做比较，但在 WRMSR 指令提交后将 DUT 当前的 MSR 值同步到 REF，确保后续依赖该 MSR 值的指令（如依赖 FS\_BASE 的内存寻址）不会因为 REF 的 MSR 值过期而产生分歧。需要注意，IA32\_EFER 和 FS/GS\_BASE 等影响指令执行语义的 MSR 仍然需要比较——它们的值由软件显式设置，DUT 和 REF 执行相同的 WRMSR 指令后应当得到相同的值。

#### 中断注入 REF 的具体方案

中断与异常的关键区别在于**触发时机**：异常是同步的，由当前正在执行的指令触发（如缺页、非法指令）；中断是异步的，由外部事件触发（如定时器到期、UART 收到数据、PLIC 转发的设备中断），与当前执行的指令无关。这个区别决定了中断在 difftest 中需要不同于异常的处理方式。

**为什么需要显式注入。** 启蒙3号的 difftest 中，DUT 和 REF 共享同一个外设模型（CLINT、PLIC 等），REF 能通过共享内存自行检测到中断信号，因此不需要 DUT 显式通知。但新模拟器使用外部共享库作为 REF（Spike、QEMU），REF 运行在独立的地址空间中，无法访问 DUT 的外设状态。如果不显式注入，REF 永远不会感知到中断的发生，两侧的执行路径会在第一次中断时彻底分歧。

**注入的时机。** DUT 的提交级是中断响应的唯一观测点。当提交级决定响应一个中断时（即当前指令正常提交后，在下一条指令开始执行之前插入中断处理），DUT 需要立即将中断号通知给 REF。时序上，这发生在"当前指令的 difftest 比较完成之后、下一条指令的 difftest 比较开始之前"。

**注入的流程。** 具体步骤如下：

1. DUT 的提交级正常提交当前指令，触发该指令的 difftest 比较（GPR、PC、CSR 等），确认当前指令的执行结果 DUT 和 REF 一致。
2. 提交级检测到有待处理的中断（中断使能且中断优先级满足条件）。DUT 执行中断处理：保存当前 PC 到 mepc/sepc（RISC-V）或压栈（x86），跳转到中断处理入口。
3. DUT 将中断号发送给 REF。REF 收到中断号后，执行自己的中断处理流程——同样保存上下文并跳转到处理入口。
4. DUT 将自己中断处理后的完整状态（包括 PC、CSR 等）同步到 REF，确保两侧在中断处理入口处的状态完全一致。这一步很重要，因为中断处理涉及多个 CSR 的同时修改（mepc、mcause、mstatus 等），DUT 和 REF 的修改顺序可能不同。
5. 从下一条指令（中断处理程序的第一条指令）开始，恢复正常的逐指令 difftest 比较。

**两种 ISA 的差异。** 中断注入的框架对两种 ISA 是相同的——都是"在指令边界注入中断号"，但细节有所不同：

- **RISC-V**：中断号对应 mcause 的值（最高位为 1 表示中断，低位为中断编码，如 7=机器定时器中断、11=机器外部中断）。中断响应修改的 CSR 包括 mepc（保存返回地址）、mcause（中断原因）、mstatus（关闭中断使能）。如果启用了中断委托（mideleg），中断可能被委托到 S 模式处理，此时修改的是 sepc、scause、sstatus。
- **x86-64**：中断号对应 IDT（中断描述符表）的向量号（如 0x20=定时器中断、0x21=键盘中断）。中断响应将 RFLAGS、CS、RIP 压栈，从 IDT 中查找处理入口并跳转。x86-64 的中断处理还涉及特权级切换（如果中断处理程序在 Ring 0 而当前在 Ring 3，需要切换栈）。

**与异常引导执行的关系。** 中断注入和异常引导执行解决不同的问题，但可以复用相同的状态同步机制。异常引导执行处理的是"同一条指令上 DUT 和 REF 检测结果不同"的问题；中断注入处理的是"REF 无法感知外部中断信号"的问题。两者都需要在注入/引导后将 DUT 的完整状态同步到 REF，这个同步步骤可以共用同一套寄存器同步机制。

#### ISA 异常的引导执行方案

异常处理是 difftest 中最容易出现不同步的场景。根本原因在于：异常的触发取决于微架构的实时状态（TLB 是否命中、页表遍历的中间结果、访存地址对齐检查等），而 DUT 和 REF 的微架构状态天然不同——DUT 是乱序流水线，REF 通常是简单的顺序解释器。同一条指令在 DUT 上可能触发缺页异常（因为 TLB 未命中），而在 REF 上可能正常执行（因为 REF 的 TLB 状态不同），或者两者触发不同类型的异常。

启蒙3号的做法是将异常简化为三个布尔标志（指令缺页、加载缺页、存储缺页），在指令提交时传递给 REF。这种方式有两个局限：一是只能表达缺页这一种异常类型，无法覆盖 x86 的通用保护异常、栈段异常、双重异常等；二是不传递异常的附加信息（如缺页地址、错误码），导致 REF 无法完整还原异常现场。

新模拟器采用 XS-GEM5 验证的**引导执行**方案：当 DUT 在提交级检测到某条指令触发了异常时，通过 `guided_exec()` 向 REF 发送一个 `ExecutionGuide` 结构体，包含以下字段：

- `force_raise_exception`：是否强制触发异常
- `exception_num`：异常编号（RISC-V 的 mcause/scause 值，x86-64 的中断向量号）
- `mtval` / `stval`：RISC-V 异常附带值（通常是触发异常的虚拟地址）
- `mtval2` / `htval` / `vstval`：Hypervisor 扩展的异常附带值（虚拟化模式下使用）
- `force_set_jump_target`：是否强制设置跳转目标
- `jump_target`：强制跳转目标地址

REF 收到引导后，在本次执行中优先采用 DUT 提供的异常信息，覆盖自己的检测结果，但仍然由 REF 自身完成异常处理的后续流程（保存上下文、设置 CSR、跳转到处理入口等）。引导执行后，通过 `regcpy()` 从 REF 读取其新状态，供后续比较使用。

需要强调的是，引导执行是一种**覆盖机制**，不是对 REF 异常检测逻辑的替代。REF（Spike、QEMU）作为外部工具以共享库方式加载，其内部的异常检测逻辑始终存在。引导执行只在 DUT 检测到异常时启用；对于不触发异常的正常指令，REF 使用自己的逻辑正常执行——如果 REF 在一条 DUT 认为正常的指令上意外触发了异常，后续的 PC/GPR 比较会立刻捕获到这个分歧，这恰好说明 DUT 漏检了某个异常，是一个需要修复的 bug。

这套机制天然支持任意异常类型和任意 ISA：

- **RISC-V 侧**：异常号对应 mcause/scause 的值（如 12=指令缺页、13=加载缺页、15=存储缺页），附带值对应 mtval/stval（通常是触发异常的虚拟地址）。启蒙3号目前只实现了约 11 种异常编码，但引导执行机制不限制异常编码的数量——未来添加新的异常类型时只需在 DUT 侧检测并传递异常号即可，无需修改引导执行逻辑本身。
- **x86-64 侧**：异常号对应中断向量号（如 13=#GP 通用保护异常、14=#PF 缺页异常、12=#SS 栈段异常），附带值对应不同异常的专属信息——缺页异常附带 CR2（触发缺页的线性地址）和错误码（指示读/写/取指、用户/内核），通用保护异常附带选择子索引等。同一套引导执行流程用异常号区分不同异常，用附带值字段承载各异常的专属信息。

对于跳转目标地址的引导也是必要的：异常处理入口地址由 DUT 的 CSR/控制寄存器决定（RISC-V 的 mtvec/stvec，x86 的 IDT），REF 的相应寄存器可能未及时同步。通过引导直接告知跳转目标，避免 REF 因控制寄存器不同步而跳到错误地址。

> **补充（spec change-20260408-013040）**：对于 RISC-V LR/SC 与 x86 LOCK CMPXCHG 等"原子事务"语义指令，`ExecutionGuide` 字段集合不变，但 `uarchstatus_cpy` / SC 成功标志的注入时机必须落在 ROB 提交"**原子事务最后一条 uOP**"那一拍——即 `last_uop=true` 的那条 uOP 的 difftest 触发点。这与 `verification-backend-api.md` §1 的 macro_id+last_uop 模型一致：原子事务的全部 uOP 必须按序连续提交，最后一条提交时同步原子状态，避免出现"中间 uOP 已写 REF 但 SC 失败"这种半提交状态。具体的 uarchstatus_cpy 调用见 `varification-design.md` §2.6.1。

#### 系统指令的 CSR 时序差异处理方案

系统指令（RISC-V 的 ecall、mret、sret、csrrw/csrrs/csrrc 系列；x86 的 INT、IRET、MOV CR、WRMSR 等）会修改控制状态寄存器。问题在于：DUT 和 REF 对"CSR 修改在何时生效"的建模可能不同。

启蒙3号之所以能直接比较全部 21 个 CSR，是因为它采用 comb/seq 分离——在比较时刻读取的是下一周期即将锁存的值，与 REF 的更新时机恰好一致。但新模拟器采用 tick() 模型，CSR 的更新时机取决于 tick() 内部的执行顺序。例如，一条 mret 指令在 DUT 的提交级将 mstatus 恢复为中断前的状态，但这个更新可能在当前 tick() 执行完毕后才对外可见；而 REF 在执行 mret 后立即更新了 mstatus。此时比较两侧的 mstatus 就会出现一个周期的偏差。

处理方案分为两层：

**第一层：识别系统指令并跳过 CSR 比较。** 当提交级发现当前指令是系统指令（会修改 CSR 的指令）时，本次比较只检查 GPR 和 PC，跳过所有 CSR 的比较。这避免了因更新时序差异导致的误报。跳过后，将 DUT 当前的 CSR 值同步到 REF，确保两侧在下一条指令时保持一致。

**第二层：维护系统指令清单。** 需要维护一份"会修改 CSR 的指令"清单（参考 XS-GEM5 的 `skipCSRs` 列表），用于判断当前提交的指令是否需要跳过 CSR 比较。XS-GEM5 的实现方式是匹配指令编码中的 CSR 字段（机器指令的 bits\[31:20]，即 12 位 CSR 编号）。对于 RISC-V，这个清单包括 ecall、ebreak、mret、sret、以及所有 CSR 读写指令；对于 x86-64，需要覆盖所有修改控制寄存器、段寄存器、标志寄存器的特权指令（MOV CR、WRMSR、LGDT/LIDT 等）。这个清单应当是可配置的，而非硬编码在比较逻辑中。

需要注意的是，跳过 CSR 比较并非放弃验证——CSR 的正确性仍然通过后续指令间接验证。如果 mret 之后的第一条指令在错误的特权级执行（因为 mstatus 恢复错误），后续的 GPR/PC 比较必然会捕获到这个错误。跳过的只是"修改 CSR 的那条指令本身"的 CSR 比较，而非永久跳过。

> **补充（spec change-20260408-013040）**：在新模拟器后端中，所有系统指令统一走 Sys/CSR pipe（参考 `verification-backend-api.md` §5），因此 `skipCSRs` 列表的实现方式建议从"按指令编码 bits[31:20] 匹配"改为**按 uOP 子类型过滤**——只要一条 uOP 的子类型属于 Sys/CSR pipe，就直接命中。该做法在双 ISA 下统一适用：RISC-V 侧 ecall/mret/sret/csrrw 等都会被前端译码为同一个 Sys/CSR uOP 子类，x86-64 侧 INT/IRET/MOV CR/WRMSR/LGDT/LIDT 同理。RV64 侧需要在 H 扩展启用时把 Hypervisor CSR（hstatus、hideleg、…）以及向量 CSR（vtype、vcsr、…）一并加入跳过逻辑。

### 4.3 综合验证策略

#### 模拟器自身正确性的两个支柱

模拟器采用高层逻辑 + 时序计数的建模方式（见 4.1），内部没有可以与外部参考直接对比的寄存器级状态。因此，模拟器自身的正确性验证依赖两个互补的支柱：

**支柱一：Difftest 保证整体功能正确性。**

通过跑程序、与参考模型（Spike/NEMU/QEMU）逐指令比较提交结果来验证（详见 4.2）。Difftest 验证的是"模拟器作为一个整体，执行程序的指令级结果是否正确"。

Difftest 的覆盖范围和局限：

- **覆盖**：所有模块的协作结果——只要某个模块的 bug 影响了指令的最终提交结果（GPR、PC、CSR、存储写入），Difftest 就能捕获。
- **局限**：不能定位 bug 到具体模块；不能检测不影响提交结果的微架构内部错误。例如分支预测器的更新逻辑有 bug，但被 ROB 保序兜住暂不影响功能——Difftest 不会报错，但会导致性能异常或在极端边界条件下触发功能故障。

**支柱二：人工编写的模块级检查保证各模块的正确性。**

这是弥补 Difftest 局限的关键手段，包括两类：

**运行时断言（assertion）**：在每个模块的 tick() 逻辑中嵌入不变量检查，与功能逻辑同步编写。断言不依赖外部参考，直接检查模块内部的约束是否成立。例如：

- "空闲物理寄存器数 + 已分配物理寄存器数 = 总数"
- "ROB 入队指针不超过出队指针一整圈"
- "发射队列中标记为就绪的操作数对应的 PRF 条目确实不忙"
- "提交的指令序列号严格递增"

断言在全系统仿真中始终运行。一旦断言失败，立即报告具体模块、违反的条件和当前状态，帮助精确定位 bug。断言应在编写 tick() 逻辑时同步编写——不是事后补充的测试，而是模块规格的一部分。

**定向 I/O 测试**：设计者为模块编写针对性的测试用例——构造特定的输入序列驱动模块运行，检查输出是否符合预期。分两个层次：

- **模块级定向测试**：将模块单独实例化（上下游用存根替代），灌入精心设计的输入，检查输出值和输出时序。例如对乘法器灌入边界值输入，检查结果是否正确且恰好在第 3 拍输出。详见 3.3 的方式三。
- **构造性验证（micro-benchmark）**：在全系统环境中运行特定的微基准测试程序，人工计算预期结果和周期数，验证实际是否匹配。例如"一串无依赖 ADD 指令在空流水线上第 N 个周期开始每周期提交一条"、"一条 DCache 缺失的 LOAD 应阻塞后续依赖指令恰好 M 个周期"。构造性验证直接检验时序行为是否符合设计者意图，不需要外部参考模型。

**两个支柱的关系**：

两者缺一不可，互相补充：

| <br />   | 支柱一：Difftest | 支柱二：模块级检查          |
| -------- | ------------ | ------------------ |
| 验证范围     | 全系统，指令级结果    | 单模块，内部不变量 + I/O 行为 |
| 参考对象     | 外部参考模型       | 无需外部参考（人工定义预期）     |
| 自动化程度    | 高（跑程序即可）     | 低（需要人工设计检查条件）      |
| bug 定位能力 | 弱（只知道哪条指令出错） | 强（精确到模块、条件、周期）     |
| 覆盖盲区     | 不影响提交结果的内部错误 | 未被检查条件覆盖的场景        |

> **对模块设计的约束**：支柱二直接约束了模块的设计方式——每个模块必须设计成容易独立调用、容易编写定向 I/O 测试和运行时断言的形式。这个可测试性要求见 3.3。

#### 各模块的完整检查方案

| Dispatch   | 特定 IQ 满   | 构造全是整数指令的 packet，而整数 IQ 已满                 | packet 停滞，尽管其他 IQ 有空                                          |
| Dispatch   | Store 拆分  | 一条 Store 指令                                | 正确拆分为 STA + STD，分别进入 IQ\_STA 和 IQ\_STD                        |
| IssueQueue | 依赖唤醒      | 两条背靠背依赖指令（B 的源 = A 的目标）                    | A 写回后，B 在下一拍标记为就绪并被发射                                         |
| IssueQueue | 最老优先      | 3 条无依赖指令同周期就绪，但只有 1 个发射端口                  | ROB ID 最老的优先发射                                                |
| IssueQueue | 多周期唤醒     | MUL（3 拍）指令后接依赖指令                           | 依赖指令恰好在 MUL 输入后第 3 拍被唤醒（不早不晚）                                 |
| IssueQueue | 队列满       | 连续灌入指令直到 IQ 满                              | Dispatch 被反压                                                  |
| ExecUnit   | ALU 延迟    | 一条 ADD 指令                                  | 输入后第 1 拍输出结果（valid=1 + 正确的数据）                                 |
| ExecUnit   | MUL 延迟    | 一条 MUL 指令                                  | 输入后第 3 拍输出结果；第 1-2 拍 valid=0                                  |
| ExecUnit   | DIV 阻塞    | DIV 执行期间灌入新指令                              | 新指令被拒绝（ready=0），直到 DIV 完成                                     |
| ExecUnit   | 分支重定向     | 分支指令执行发现误预测                                | 产生 redirect 信号（重定向 PC + 分支 tag）                               |
| PRF        | 旁路转发      | 背靠背依赖指令（A→B，延迟 1 拍）                        | B 从旁路网络获取 A 的结果，无需等待寄存器文件写入                                   |
| PRF        | 多分支仲裁     | 同周期两条分支都报告误预测                              | 只有 ROB ID 更老的分支生效，产生唯一的重定向信号                                  |
| ROB        | 满阻塞       | 连续分配直到 ROB 满                               | Dispatch 被反压；新指令停滞                                            |
| ROB        | 异常提交      | 异常指令前有 N 条正常指令                             | N 条正常指令全部提交后，异常指令触发 flush                                     |
| ROB        | 同周期完成     | 多条指令在同一周期完成                                | 所有完成标记正确更新，组提交在整行完成后触发                                        |
| ROB        | 中断响应      | 在两条指令之间注入中断                                | 前一条指令正常提交后响应中断，进入中断处理                                         |
| LSU        | STLF 命中   | Store x1,\[addr]; Load x2,\[addr] 同地址      | Load 从 STQ 转发获取 Store 的数据，不访问 DCache                          |
| LSU        | STLF 未就绪  | Store 地址已知但数据未就绪，后续 Load 同地址               | Load 进入 Retry 状态，等待 Store 数据就绪后重新发射                           |
| LSU        | 队列满       | 连续 Store 指令填满 STQ                          | Dispatch 停止分派新的 Store 指令                                      |
| LSU        | 误预测恢复     | 分支误预测，误预测路径上有若干 Load/Store                 | 未发送的 Load 立即释放；已发送的 Load 标记 killed，等回包后释放；误预测路径的 STQ 条目回收     |

**存储子系统**

| 模块     | 测试场景        | 输入构造                 | 预期行为                                       |
| ------ | ----------- | -------------------- | ------------------------------------------ |
| DCache | 连续命中        | 访问已在 cache 中的地址      | 1 拍返回数据                                    |
| DCache | Cache miss  | 访问不在 cache 中的地址      | 分配 MSHR，发起 refill 请求，延迟 N 拍后返回             |
| DCache | 脏行替换        | 写入地址导致 dirty 行被替换    | 先将 dirty 行写回下一级，再执行 refill                 |
| DCache | MSHR 满      | 短时间内大量 miss          | 后续 miss 请求被阻塞直到有 MSHR 释放                   |
| PTW    | TLB miss 遍历 | 访问不在 TLB 中的虚拟地址      | 发起正确级数的页表遍历（SV39 三级、PML4 四级），最终 refill TLB |
| PTW    | 无效页表项       | 遍历过程中遇到 valid=0 的页表项 | 报告 Page Fault，不继续遍历                        |
| PTW    | 并发 miss     | 两个不同虚拟地址同时 TLB miss  | 遍历请求排队或并行处理，互不干扰                           |

##### 三、构造性验证（全系统 Micro-benchmark）

以下测试在完整的全系统仿真环境中运行特定程序，验证跨模块协作的时序正确性。

| 测试场景           | 程序描述                                       | 预期时序                                                 | 验证目标                                   |
| -------------- | ------------------------------------------ | ---------------------------------------------------- | -------------------------------------- |
| 稳态吞吐量          | 空流水线，一串无依赖 ADD 指令（数量 >> 流水线深度）             | 流水线充满后，每周期提交 COMMIT\_WIDTH 条指令                       | 流水线各级的前传和提交带宽                          |
| 数据旁路延迟         | 链式依赖序列：x1←x0+1; x2←x1+1; x3←x2+1; ...      | 每条指令在前一条完成后 1 拍发射（旁路延迟 = 0 或 1 拍）                    | PRF 旁路网络 + IssueQueue 唤醒时序             |

| STLF 延迟        | Store \[addr],x1; Load x2,\[addr]（紧邻）      | Load 从 STQ 获取数据的延迟拍数                                 | LSU 内部的 Store-to-Load Forwarding 时序    |

| IQ 满阻塞恢复       | 大量发往同一 IQ 的指令（如全是整数 ALU 指令）填满该 IQ          | IQ 满后 Dispatch 对该类型指令停滞；指令发射后 IQ 释放空间                | IQ 满 → Dispatch 选择性反压 → IQ 释放 → 恢复     |
| 中断响应延迟         | 一串 NOP 指令中间注入定时器中断                         | 中断在当前提交指令后的固定拍数内被响应                                  | 提交级中断检测 → flush → 跳转到中断入口的延迟           |
| 多级页表遍历延迟       | 访问一个 TLB miss 的地址                          | 从 TLB miss 到数据返回的总延迟 = PTW 遍历延迟（每级一次内存访问）+ DCache 延迟 | PTW → Memory → TLB refill → 重试访存的完整路径  |

#### 启蒙3号作为参考的适用范围

新模拟器后端以 uOP 为调度粒度，与启蒙3号（RISC-V 指令粒度）存在根本性差异，导致后端内部状态不可直接对比。但前端和存储子系统不受影响。

| 对比类别                 | 是否可行       | 说明                                     |
| -------------------- | ---------- | -------------------------------------- |
| 指令提交结果               | **可精确对比**  | 功能语义不变，difftest 完全有效                   |
| 前端行为（BPU、ICache、FTQ） | **可精确对比**  | 前端不受 uOP 拆分影响                          |
| 缓存/TLB 行为            | **可精确对比**  | 存储子系统不关心请求来自指令还是 uOP                   |
| IPC                  | **可对比趋势**  | 绝对值因 uOP 开销会有偏差，但量级应一致；偏差超过 2 倍可能是 bug |
| 后端内部状态（ROB、IQ、PRF）   | **不可对比**   | 粒度根本不同，对比无意义                           |
| 指令提交周期               | **有系统性偏差** | 可容忍可预期的延迟差，但不能作为精确基线                   |

### 4.4 Task2 对验证接口的影响

并行的 Task2（模拟器功能完善、Ref 改造）有两个与验证直接相关的架构演进：

1. **RefCpu 统一为 librefcpu.a**：通过稳定 API（`init/reset/step`、`get_state/set_state`）调用。新模拟器的 difftest 接口设计应与之对齐，避免集成障碍。
2. **内存空间离散化**：IO 空间和内存空间显式区分。difftest 中的存储比较需要区分内存写入和 MMIO 写入（MMIO 有副作用且不可重复）。

## 五、模拟器作为 REF：RTL 验证框架

当 LLM 或 BSD 生成了 RTL 后，需要验证其正确性。此时模拟器的角色反转——从 DUT 变为 REF（golden reference）。

RTL 验证包含两个层次：**整体指令级 difftest**（验证 RTL 整个 CPU 的提交结果）和**模块级联合仿真**（验证 RTL 单个模块的周期级 I/O 行为）。前者大量复用第四章的基础设施，后者是本章的重点。

### 5.1 整体指令级 difftest：复用 DUT 框架

整体指令级 difftest 与第四章的 DUT 方向几乎对称——模拟器编译为 .so，RTL 仿真程序通过 `dlopen` 加载，每条指令提交时比较状态。

**可直接复用第四章的部分**：

| 组件              | 复用方式                                     |
| --------------- | ---------------------------------------- |
| difftest API 签名 | 完全复用，方向参数反转                              |
| ISA 无关状态容器      | 完全复用                                     |
| Proxy 工厂模式      | 模式复用（模拟器自身被编译为 .so 供外部加载）                |
| 特殊处理逻辑          | 大部分复用（MMIO 跳过、中断注入、异常引导等机制相同，"谁信任谁"方向不同） |
| 比较范围            | 完全复用（GPR、PC、CSR、存储写入）                    |

**需要额外开发**：

| 组件            | 说明                                                                                                       |
| ------------- | -------------------------------------------------------------------------------------------------------- |
| 模拟器 .so 编译目标  | Makefile 增加共享库编译目标。处理符号导出（`-fvisibility=hidden` + 显式标注）和符号隔离（`dlmopen(LM_ID_NEWLM, ...)` 防冲突）。           |
| uOP 聚合（RTL 侧） | DUT 方向的聚合在模拟器内部；REF 方向的聚合在 RTL 侧——如果 RTL 也是 uOP 架构则由 RTL 提交逻辑聚合后触发 difftest，如果 RTL 是指令级提交（如 BSD 生成）则不需要。 |

### 5.2 模块级联合仿真：核心增量能力

整体 difftest 只能验证"RTL 整个 CPU 的提交结果是否正确"，无法定位 bug 在哪个模块。模块级联合仿真将验证粒度从整个 CPU 下沉到单个流水线模块——这是本模拟器相比传统参考模型（Spike/NEMU）的核心增量价值。

| 方面        | 整体指令级 difftest | 模块级联合仿真                   |
| --------- | -------------- | ------------------------- |
| 验证对象      | RTL 整个 CPU     | RTL 单个模块                  |
| 比较粒度      | 指令提交后的架构状态     | 每周期的模块 I/O                |
| 能否定位 bug  | 只能定位到"哪条指令出错"  | 可定位到"哪个模块、哪个周期、哪个端口出错"    |
| 前提        | RTL 已组装为完整 CPU | RTL 单个模块即可验证              |
| 参考标准      | 模拟器整体执行结果      | 模拟器单个模块的 tick() 输出        |
| 复用 DUT 框架 | 高              | 低（Bridge、Trace、端口映射全部新开发） |

两者互补：模块级联合仿真用于**开发阶段**逐模块验证 LLM/BSD 生成的 RTL；整体 difftest 用于**集成阶段**验证组装后的完整 RTL CPU。

#### 比较边界：严格限定在模块 I/O

模块级联合仿真的比较**严格限定在模块的输入输出端口**，不涉及模块内部状态的对比。

这不是一个可选的设计取舍，而是建模层次决定的必然约束（见 4.1）：模拟器用高层逻辑 + 时序计数实现功能，RTL 用寄存器级流水线实现同一功能，两者的内部状态组织方式根本不同。例如一个 3 拍乘法器，模拟器内部是"结果 + 计数器"，RTL 内部是 3 级流水线寄存器——比较内部状态没有意义。

但 I/O 层比较足以验证时序正确性。如果 C++ 模块在输入后第 3 个周期在输出端口置 valid=1 并输出结果，而 RTL 在第 5 个周期才输出，第 3 个周期的比较就会发现 valid 信号不一致并报错。反之，如果两侧在每个周期的输出端口值都一致，则 RTL 的 I/O 时序行为与模拟器完全匹配——这正是"模块级周期精确"的验证目标。

#### 实现层次

#### Bridge 层：单个模块的 RTL 替换验证

在全系统仿真中，某个 C++ 模块与 Verilator 编译的 RTL 模块并行运行，通过 Bridge 层桥接。Bridge 每个周期执行以下步骤：读取上游输入 → 转换为 RTL 信号灌入 → 两侧分别执行 → 读取并转换 RTL 输出 → 比较两侧输出 → 将参考输出写入下游。

Bridge 层的类设计和伪代码见 `varification-design.md` 第三章。

#### Trace 录制与回放

在全系统仿真时录制每个模块的输入/输出 trace，后续可以：

- 用录制的输入 trace 独立驱动一个模块运行，验证其输出。
- 用录制的 trace 驱动 Verilator RTL 模块，与录制的参考输出比较。
- 分析 trace 定位 bug（哪个周期、哪个信号最先出现偏差）。

### 5.3 DPI-C 通信机制

Verilator 联合仿真中，C++ 和 RTL 之间的通信通过 DPI-C（Direct Programming Interface for C）实现。DPI-C 是 SystemVerilog 标准的一部分，允许 Verilog 模块调用 C 函数，也允许 C 程序读写 Verilog 信号。

在我们的框架中，DPI-C 主要用于两个场景：

1. **调试信号导出**：RTL 模块将指令提交信号（PC、写回寄存器号/数据等）通知给 C++ 仿真驱动程序。
2. **模块级端口桥接**：Bridge 层通过 Verilator 生成的 C++ 类直接读写 RTL 模块的输入/输出引脚，不需要显式 DPI-C 函数。

具体的 DPI-C 接口设计和代码示例见 `varification-design.md` 第三章。

### 5.4 端口映射规范

模块级联合仿真的关键技术挑战是：如何将模拟器的逻辑端口（如"一条 uOP 结构体"）映射为 RTL 的信号级端口（如一组位宽确定的信号线）。

我们采用**声明式端口定义**：每个端口在 C++ 中声明时同时携带逻辑语义和信号映射信息。这种设计确保：

- LLM 阅读端口声明就能理解接口语义（场景2）。
- Bridge 层能自动生成 C++ ↔ Verilator 的信号映射代码。
- 端口声明是 uOP 字段定义的唯一依赖点——uOP 设计确定后只需更新端口声明，框架其余部分不受影响。

具体的端口声明结构体设计和信号映射规范见 `varification-design.md` 第三章。

### 5.5 BSD 位级展开层

如需支持 BSD 生成 RTL，需要将模拟器的高层端口拍平为 bool 数组（BSD 的输入/输出格式）。BSD 训练用 10 万组随机输入采样模拟器的位级 I/O，验证用 1 亿组输入对比。展开层可由端口定义中的信号级映射信息自动生成。

### 5.6 差异报告

模块级比较的差异报告需要比指令级更详细——不仅报告"哪个周期不一致"，还要报告"哪个输出端口、哪些位不一致"。

参考 XS-GEM5 的 `reportDiffMismatch()` 模式，差异报告应包含：

- **逐字段差异标记**：对 GPR×32、FPR×32、CSR 集合、向量寄存器逐项显示 REF 值和 DUT 值，差异项高亮标记
- **指令历史**：转储最近 N 条（如 20 条）已提交指令的反汇编文本，帮助定位出错上下文
- **双侧完整状态**：同时调用 REF 的 `isa_reg_display()` 和 DUT 侧的寄存器显示，提供完整的对比视图
- **PC 和指令信息**：当前 PC、指令编码、特权级等基本执行上下文

### 5.7 分阶段实现路径

**Phase 1：DUT + REF 指令级 difftest**

- DUT 方向（第四章）：集成 Spike 或 librefcpu.a
- REF 方向（5.1）：模拟器增加 .so 编译目标，暴露 difftest API
- 重点：uOP→指令边界聚合、ISA 无关状态容器

**Phase 2：模块级联合仿真框架**

- Bridge 层骨架（5.2）、Trace 录制/回放（5.2）、端口映射规范（5.4）
- 用一个简单模块（如 ALU）跑通"C++ 模块 ↔ Verilator RTL"的闭环

**Phase 3：ISA 无关 + x86-64 支持**

- `DifftestState` 替换硬编码结构，引入 RegClass 注册
- 封装 QEMU x86-64 TCG 为 .so，实现 x86-64 Proxy + 特殊处理（RFLAGS 掩码、REP 前缀、段寄存器、x87 FPU 栈）

## 六、术语表

| 术语                         | 英文                                 | 含义                                                                                       |
| -------------------------- | ---------------------------------- | ---------------------------------------------------------------------------------------- |
| **tick**                   | tick                               | 一个仿真周期的推进动作。每个模块在每个周期被调用一次 tick() 方法，完成该周期的全部逻辑。                                         |
| **周期精确**                   | cycle-accurate                     | 仿真的精度级别：每个仿真周期的行为精确对应真实硬件的一个时钟周期。区别于"指令精确"（只关心最终结果）或"近似精确"（只保证统计意义上的近似）。                 |
| **uOP / 微操作**              | micro-operation                    | ISA 无关的基本操作单元。前端译码器将不同 ISA 的指令翻译为统一的 uOP，后端以 uOP 为粒度进行调度和执行。                             |
| **nOP**                    | nano-operator                      | AGON 论文中的概念，一组基本操作符（如 REG\_READ、ADD、MEM\_WRITE），用于描述指令的功能。与 uOP 精神类似但粒度更细。               |
| **SimObject / SimModule**  | SimObject / SimModule              | 所有仿真组件的基类，提供统一的生命周期管理（创建、初始化、端口连接、运行）和层次化命名。                                             |
| **Port（端口）**               | Port                               | 模块间通信的标准接口。分为 RequestPort（发起请求方）和 ResponsePort（响应方），通过 bind() 建立连接。                      |
| **bind（绑定）**               | bind                               | 将两个端口连接在一起的操作。绑定后双方可以互相通信。绑定是双向的——一次 bind 同时建立两个方向的连接。                                   |
| **TimeBuffer（时间缓冲区）**      | TimeBuffer                         | 流水线级间传递数据的通道，带固定延迟。上游写入数据后，下游在 N 个周期后才能读到，模拟流水线寄存器的行为。                                   |
| **trace（轨迹录制）**            | trace                              | 在仿真过程中录制某个模块的输入/输出数据序列。录制的 trace 可以用于后续的回放验证或调试分析。                                       |
| **stub（存根）**               | stub                               | 只声明了接口但不做实际计算的占位模块。用于在独立测试某个模块时替代其上下游模块，提供最简输入或无条件接受输出。                                  |
| **bridge（桥接层）**            | bridge                             | C++ 模拟器模块与 Verilator RTL 模型之间的转换层。负责将逻辑级数据类型映射为信号级位宽，以及同步两侧的时钟。                          |
| **golden reference（黄金参考）** | golden reference                   | 被认为是正确的参考实现。在 RTL 验证中，C++ 模拟器模块就是 golden reference，LLM 生成的 RTL 与之对比来判断正确性。               |
| **Verilator**              | Verilator                          | 一个开源工具，能将 Verilog/SystemVerilog 代码编译为 C++ 仿真模型。编译后的模型可以在 C++ 程序中实例化和驱动，用于高速仿真和验证。        |
| **DUT（被测设备）**              | Device Under Test                  | 正在被验证的模块。在 RTL 验证场景中，LLM 生成的 RTL 就是 DUT，模拟器的 C++ 模块是参考。                                  |
| **checkpoint（检查点）**        | checkpoint                         | 仿真运行到某个时间点时保存的完整状态快照。可以从 checkpoint 恢复仿真，避免每次从头开始。                                       |
| **ISA（指令集架构）**             | Instruction Set Architecture       | 定义处理器可以执行哪些指令、寄存器有多少、寻址方式是什么等。常见 ISA 有 x86、RISC-V、ARM 等。                                 |
| **RTL（寄存器传输级）**            | Register Transfer Level            | 用 Verilog 或 VHDL 描述硬件电路的方式，精确到每个寄存器和逻辑门。是芯片制造前的设计输入。                                     |
| **DSE（设计空间探索）**            | Design Space Exploration           | 在给定的设计模板内自动搜索最优参数组合的方法。通常通过遍历或启发式搜索不同的配置，运行仿真获取性能数据来实现。                                  |
| **comb / seq**             | combinational / sequential         | 启蒙3号模拟器的编程风格，将每个模块的逻辑分为组合逻辑（comb，不更新状态）和时序逻辑（seq，锁存状态）两个阶段。                              |
| **ROB**                    | Reorder Buffer                     | 重排序缓冲区，乱序处理器中按程序顺序维护指令的关键结构。保证乱序执行的结果能按正确的程序顺序提交。                                        |
| **IPC**                    | Instructions Per Cycle             | 每周期提交指令数，衡量处理器性能的核心指标。                                                                   |
| **difftest（差分测试）**         | differential testing               | 将两个实现（如乱序 CPU 和简单参考模型）跑同一个程序，逐条比较结果是否一致。启蒙3号已有此机制。                                       |
| **PortBundle（端口束）**        | PortBundle                         | 一组相关端口的集合。比如 Rename 模块的输入端口束可能包含"待重命名的指令组"和"空闲寄存器列表"等多个端口。                               |
| **DPI-C**                  | Direct Programming Interface for C | SystemVerilog 标准的一部分，允许 Verilog 模块调用 C 函数，也允许 C 程序读写 Verilog 信号。是 RTL 与 C++ 仿真程序通信的主要机制。 |
| **dlopen / dlsym**         | dynamic loading                    | Linux 动态链接库加载机制。`dlopen` 在运行时加载 .so 文件，`dlsym` 获取其中的函数指针。difftest 中用于加载参考模型共享库。          |
| **store commit queue**     | store commit queue                 | 香山 NEMU 的特性——记录每条存储指令提交时的地址、数据和写掩码，与 RTL 侧的存储提交逐一比较，检测内存写入错误。                            |
| **guided execution（引导执行）** | guided execution                   | 香山 NEMU 的高级特性——DUT 可以引导参考模型的异常处理和跳转行为，解决参考模型中的非确定性问题。                                    |
| **snapshot（快照）**           | snapshot                           | 导出模块当前内部状态为结构化数据的操作。用于 trace 录制、调试分析和检查点保存。每个 SimModule 需要实现此接口。                         |
| **assertion（断言）**          | assertion                          | 嵌入在模块内部的运行时检查，验证不变量（如"空闲寄存器数 + 已分配寄存器数 = 总数"）。最轻量的内部验证方式。                                |
| **共享库（.so）**               | shared library                     | Linux 下的动态链接库格式。模拟器编译为 .so 后，外部 RTL 仿真程序可以在运行时加载并调用其中的 difftest 函数。                      |
