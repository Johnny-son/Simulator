# 关键组件与模块边界

## 1. 目标

定义模块边界的核心目的：

1. 防止跨层耦合。
2. 支持多 ISA 插件化接入。
3. 支持单模块替换验证（C++/RTL）。

---

## 2. 前端组件边界

### 2.1 `BPU`

职责：
- 只做预测，不生成语义 uop。

输入：
- PC、历史、训练事件。

输出：
- `predict_dir/predict_target/meta`。

### 2.2 `IFetch/ICache/ITLB`

职责：
- 只提供指令字节流与 fault 信息。

禁止：
- 做 ISA 语义解释。

### 2.3 `PreDecode`

职责：
- 切分指令边界、轻判控制流类型。

输出：
- `DecodedInstSeed`。

### 2.4 `DecodeRISCV` / `DecodeX86`（插件点）

职责：
- 将 ISA 指令翻译成统一 `FrontUop`。

边界规则：
1. 可读取 ISA 特有字段。
2. 不可依赖后端内部结构。

### 2.5 `FrontEnqueue`

职责：
- 协议适配，送入 `UopBuffer`。

禁止：
- 重新解释 uop 语义。

---

## 3. 后端组件边界

### 3.1 `UopBuffer`

职责：
- 前后端解耦。
- 管理 `head/tail + epoch`。
- flush 时执行 younger kill。

对外接口：
- 仅暴露通道与事件接口，不暴露内部数组。

### 3.2 `Rename`

职责：
- RAT/FreeList/checkpoint。

禁止：
- 直接操作 ROB 内部存储。

### 3.3 `Dispatch + IssueQueue`

职责：
- 分发、就绪跟踪、发射仲裁。

禁止：
- 修改 `archRAT` 或提交架构态。

### 3.4 `Exu`

职责：
- 非访存执行。

输出：
- 结果 + 完成位来源。

### 3.5 `Lsu`

职责：
- 访存顺序与原子事务。

禁止：
- 越过 Commit 直接更新架构可见状态。

### 3.6 `WriteBack`

职责：
- 汇合完成结果，驱动 wakeup。

### 3.7 `ROB + Commit`

职责：
- 顺序退休。
- 全机恢复仲裁。

规则：
- 最终 `FlushEvent` 只在此处产出。

---

## 4. 多 ISA 插件架构（新增）

## 4.1 接口分层

1. `IsaFrontendAdapter`
- 负责某一 ISA 的 predecode/decode 实现。

2. `DecoderRegistry`
- 管理多个 `IsaFrontendAdapter` 实例。

3. `Common Front Pipeline`
- 不关心具体 ISA，仅按 `IsaTag` 调用 registry。

### 4.2 推荐接口

```cpp
class IsaFrontendAdapter {
public:
  virtual ~IsaFrontendAdapter() = default;
  virtual IsaTag isaTag() const = 0;
  virtual int decode(const DecodedInstSeed& in, FrontUop out[4]) const = 0;
};

class DecoderRegistry {
public:
  void registerAdapter(IsaFrontendAdapter* adapter);
  const IsaFrontendAdapter* find(IsaTag tag) const;
};
```

### 4.3 新 ISA 接入清单

1. 增加 `IsaTag` 枚举值。
2. 实现 `IsaFrontendAdapter`。
3. 注册到 `DecoderRegistry`。
4. 完成该 ISA 的最小定向测试：
- 线性指令
- 分支
- 访存
- 异常/系统指令

---

## 5. 指针与状态所有权边界

1. `UopBuffer` 指针只归 `UopBuffer`。
2. `ROB` 指针只归 `ROB`。
3. `IQ/LSQ` 指针只归各自 owner。
4. 跨模块只能传“索引值与事件”，不能传“可写状态引用”。

---

## 6. 模块边界硬规则

1. 前端不访问后端内部状态。
2. 后端不依赖 ISA 原始编码。
3. 边界字段统一 `wire/reg`。
4. 同名字段全局同义（`seq_id/rob_idx/br_mask/macro_id`）。
5. Flush 由统一事件总线传播，禁止私有 flush 通道各自解释。

---

## 7. 验证友好性要求

每个模块必须具备：

1. 可独立实例化。
2. 可序列化状态。
3. 可插桩观测关键统计。
4. 可挂接 trace 录制/回放。

这四项是后续 RTL 替换验证的前置条件。
