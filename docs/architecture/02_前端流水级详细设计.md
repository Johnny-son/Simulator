# 前端流水级详细设计

## 1. 总体职责

前端负责：

1. 取指路径控制（预测、请求、返回、切分）。
2. 多 ISA 指令识别与译码。
3. 生成统一 `CommonUopPacket` 并送入后端边界。

## 2. 流水级定义

### 2.1 `PIFetch`

职责：

- 选择 `next_pc`（顺序 PC / redirect PC / 异常入口）。
- 发起分支预测，生成预测元信息。
- 发送取指请求（可接 ITLB/权限校验入口）。

输入：

- 来自后端的 `FlushBus`（可能带 `redirect_pc`）。
- 上拍预测状态。

输出：

- `fetch_req`（PC、请求 ID、预测 meta）。

### 2.2 `IFetch`

职责：

- 接收 ICache 返回的指令块。
- 传递异常信息（如取指页故障/访问异常）。

输入：`fetch_resp`。

输出：`inst_block`（包含指令字节流、pc_base、异常位）。

### 2.3 `PreDecode`

职责：

- 变长指令切分与对齐。
- 控制流轻判（branch/jump/call/ret）。
- 形成 decode 所需的基础标签。

输入：`inst_block`。

输出：`predecode_packet`（一组已切分指令 + 轻判信息）。

### 2.4 `Decode`

职责：

- 按 ISA 类型进入对应译码器（如 `DecodeRISCV` / `DecodeX86`）。
- 生成统一 `CommonUop`。
- 附带预测相关字段用于后续校验。

输入：`predecode_packet`。

输出：`CommonUopPacket`。

### 2.5 `FrontEnqueue`

职责：

- 将 `CommonUopPacket` 按 `valid/ready` 协议送入边界队列（`UopBuffer`）。

输入：`CommonUopPacket`。

输出：`fe2be Channel<CommonUopPacket>`。

## 3. 前端关键约束

1. `valid=1 && ready=0` 时，`bits` 必须保持稳定。
2. 收到 flush 的当拍，清除 younger 指令的有效位。
3. 下一拍从 `redirect_pc` 重启取指。
4. 前端不直接依赖 ROB 结构，依赖统一的 `FlushBus`。
