# 前端流水级详细设计

## 1. 设计目标

前端的目标不是“把指令尽快送出去”，而是提供一个**可恢复、可扩展、可验证**的统一入口：

1. 支持多 ISA 输入（RISC-V、x86，后续可扩展）。
2. 输出 ISA 无关的 `FrontUop`。
3. 与后端通过统一握手协议解耦（`FrontEnqueue -> UopBuffer -> Rename`）。
4. 在 flush/redirect 下提供精确恢复语义。

---

## 2. 前端时序契约（必须遵守）

前端与后端统一采用 `comb_begin -> comb -> seq` 三段式。

1. `comb_begin`
- 清空本拍组合输出。
- 复制本级时序态到工作副本（若该级有状态）。

2. `comb`
- 只读取：上拍寄存态 + 上游输入 + 广播事件。
- 只写入：本拍组合输出 + 工作副本。

3. `seq`
- 统一锁存工作副本，成为下一拍寄存态。

强约束：
- 禁止在同一拍内让“下游 ready 再反向决定上游有效数据内容”。
- `valid=1 && ready=0` 时，`bits` 必须保持稳定，不允许改写。

---

## 3. 流水级定义（5 级）

### 3.1 `PIFetch`

职责：
- 维护取指 PC。
- 处理 `FlushEvent` 的 `redirect_pc`。
- 生成取指请求与预测上下文。

输入：
- `FlushEvent`（来自后端仲裁点）。
- 上拍局部状态（当前 fetch PC、重定向锁存位）。

输出：
- `PIFetchIFetchIO`（指令块请求上下文/元数据）。

状态（建议）：
- `pc_q` / `pc_n`
- `redirect_pending_q` / `redirect_pending_n`

### 3.2 `IFetch`

职责：
- 对接 ICache/ITLB（首版可 stub）。
- 返回指令块、PC、fault 标志。

输入：
- `PIFetchIFetchIO`
- 下游 `ready`

输出：
- `FrontDecIO`

关键约束：
- `ready=0` 时保持输出 lane 的 `inst/pc/valid/predict_meta` 稳定。

### 3.3 `PreDecode`

职责：
- 指令边界切分（变长 ISA 时必需）。
- 轻量控制流识别（branch/call/ret hint）。
- 为 Decode 提供 ISA 识别输入。

输入：
- `FrontDecIO`

输出：
- `PredecodePacket`（可与 `FrontDecIO` 合并为一个结构，语义需固定）。

### 3.4 `Decode`

职责：
- 选择具体 ISA 解码器。
- 生成统一 `FrontUop`。
- 补齐 `is_microop/is_last_microop/macro_id` 等边界字段。

输入：
- `PredecodePacket`
- 下游 `ready`

输出：
- `DecEnqIO`

### 3.5 `FrontEnqueue`

职责：
- 将 `FrontUop` 送入 `UopBuffer`。
- 统一处理 backpressure 与 flush kill。

输入：
- `DecEnqIO`
- `UopBuffer.ready`
- `FlushEvent`

输出：
- `fe2be`（`ChannelBundle<FrontUop, FE_BE_WIDTH>`）

## 4. 多 ISA 插件化接口

### 4.1 设计原则

1. 前端允许 ISA 感知，后端禁止 ISA 分支。
2. 新增 ISA 时，不修改主流水时序，只注册新解码器。
3. ISA 特有字段不直接泄漏到 `ExecUop`。

### 4.2 接口分层

1. `IIsaDetector`
- 输入：原始字节流与前缀上下文。
- 输出：`IsaTag` + 指令长度。

2. `IPreDecoder`
- 输入：指令块。
- 输出：按 lane 切分后的 `DecodedInstSeed`。

3. `IIsaDecoder`
- 输入：`DecodedInstSeed`。
- 输出：`FrontUop` 或 `FrontUop[]`（微码分解场景）。

4. `DecoderRegistry`
- `registerDecoder(IsaTag, IIsaDecoder*)`
- `decode(IsaTag, seed)`

### 4.3 推荐伪代码

```cpp
struct DecodedInstSeed {
  wire<64> pc;
  wire<32> raw_inst_lo;
  wire<32> raw_inst_hi; // 变长 ISA 可用
  wire<2>  isa_tag;
  wire<1>  predict_taken;
};

class IIsaDecoder {
public:
  virtual ~IIsaDecoder() = default;
  virtual int decode(const DecodedInstSeed& in, FrontUop out[4]) const = 0;
};
```

## 5. 前后端边界协议

边界固定为：

`Decode -> FrontEnqueue -> UopBuffer -> Rename`

其中：
- `FrontEnqueue` 只负责协议与节流，不负责语义再解释。
- `UopBuffer` 拥有独立 ring 指针（`head/tail + epoch`），是前后端唯一节流缓冲。
- flush 时由 `UopBuffer` 执行 younger 清理（按统一年龄比较函数）。

---

## 6. flush/redirect 前端响应规范

1. 当拍收到 `FlushEvent.valid=1`：
- 所有前端级间输出 `valid` 清零（kill younger）。
- 停止将旧路径数据继续前推。

2. 下一拍：
- `PIFetch` 从 `redirect_pc` 重新取指。

3. 并发场景：
- flush 与 backpressure 同拍时，flush 优先。
- flush 与正常 fire 同拍时，最终以 flush 结果为准。

---

## 7. 观测与验证建议

建议最少提供以下可观测项：

1. `frontend_stall_cycles`
2. `frontend_flush_kill_uops`
3. `decode_uop_count_by_isa`
4. `decode_microop_ratio`
5. `redirect_recovery_cycles`

最小测试集：

1. 线性无分支程序（验证 steady-state 吞吐）。
2. 分支命中与错判（验证 redirect）。
3. `ready=0` 长背压（验证 `bits` 稳定性）。
4. flush 与背压并发。
5. RISC-V 与 x86 混合输入（先用 stub 指令流）。

---

## 8. 落地顺序建议

1. 先冻结前端级间结构体字段名和握手语义。
2. 接入 `UopBuffer`，打通单 ISA（RISC-V）闭环。
3. 引入 `DecoderRegistry` 与 x86 stub 解码器。
4. 最后再做预译码复杂功能（变长切分、宏融合、REP 展开）。
