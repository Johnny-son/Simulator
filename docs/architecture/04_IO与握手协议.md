# IO 与握手协议

## 1. 统一协议

建议全流水级使用统一通道模型：

`Channel<T> { valid, ready, bits }`

语义：

- `fire = valid && ready`
- `valid=1 && ready=0` 时，发送端必须保持 `bits` 不变
- `ready=0` 表示下游背压，必须向上游传播

### 1.1 类型系统约束（强制）

为确保前后端统一，跨模块 IO 字段统一使用 `wire<N>/reg<N>`：

1. 结构体字段必须显式位宽。
2. 不允许在边界协议中使用裸 `bool/int/uint32_t`。
3. 常量与枚举对应位宽必须有静态检查。

## 2. 宽度化通道（多发射）

实际流水按 lane 传输，采用 bundle 约定：

`ChannelBundle<T, W> { valid[W], bits[W], ready }`

语义：

1. `fire[i] = valid[i] && ready`。
2. `ready` 对全 bundle 生效（首版简化）；后续可扩展 per-lane ready。
3. 若 `valid[i]=1 && ready=0`，该 lane 的 `bits[i]` 必须稳定。

## 3. 前后端边界通道

边界建议固定：

`FrontEnqueue -> UopBuffer -> Rename`

### 3.1 通道定义（概念）

- `fe2be`: `ChannelBundle<FrontUop, FE_BE_WIDTH>`
- `be2fe_flush`: `Channel<FlushEvent>`（或独立 `FlushBus`）

`FrontUop` 与 `ExecUop` 分层：

- `FrontUop`：decode 输出语义。
- `ExecUop`：rename 后执行语义（preg/rob/完成掩码等）。

## 4. 推荐数据结构

### 4.1 `FrontUop` 最小字段

- 标识：`uop_id`, `seq_id`, `pc`, `isa_tag`
- 运算：`op_class`, `sub_op`
- 操作数：`src[]`, `dst`, `imm`
- 控制：`is_branch`, `is_call`, `is_ret`, `pred_taken`, `pred_target`, `br_id`, `br_mask`
- 异常：`except_valid`, `except_code`

### 4.2 `ExecUop` 增量字段

- `src_preg[]`, `dst_preg`, `old_dst_preg`
- `rob_idx`, `rob_epoch`
- `expect_mask`, `cplt_mask`
- 调试侧带：`dbg_meta`, `tma_meta`

### 4.3 `FlushEvent` 最小字段

- `valid`
- `cause`（mispredict/exception/replay/fence）
- `flush_seq_id`（语义定义需配合回卷年龄比较函数）
- `redirect_pc`

### 4.4 分支恢复广播字段

- `br_id`
- `br_mask`
- `clear_mask`（清除已解析分支 bit）

## 5. 队列策略

- `UopBuffer` 初始深度建议 8~16。
- 满队列时 `fe2be.ready=0`。
- 后续可增加高低水位统计与自适应节流。

## 6. 禁止事项（避免接口漂移）

1. 禁止在前端/后端各自维护不同握手语义。
2. 禁止在不同模块定义同名异义字段（如 `seq_id` 口径不一致）。
3. 禁止绕过通道直接读取其他模块内部状态。
