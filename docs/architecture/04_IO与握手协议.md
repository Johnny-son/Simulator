# IO 与握手协议

## 1. 目标

本文件用于冻结跨模块通信协议，避免“同名字段不同语义”与“握手漂移”。

协议目标：

1. 同一协议可覆盖前端、后端、验证层。
2. 支持多发射宽度与可扩展 lane。
3. 支持 flush/backpressure 并发时的确定性行为。

---

## 2. 统一时序语义

全模块统一三段时序：

1. `comb_begin`：清空输出、复制工作副本。
2. `comb`：计算本拍组合输出。
3. `seq`：锁存下一拍状态。

协议解释基于“拍边界采样”：

- `valid/ready/bits` 的生效点在拍末采样。
- 当 `valid=1 && ready=0` 时，发送端必须保持 `bits` 稳定直到握手成功。

---

## 3. 通道模型

### 3.1 标准单通道

`Channel<T> { valid, ready, bits }`

规则：

1. `fire = valid && ready`
2. `valid=0` 时 `bits` 可为 don't care（建议清零便于 debug）
3. `ready` 由消费端产生，必须可反压上游

### 3.2 宽度化通道

`ChannelBundle<T, W> { valid[W], ready, bits[W] }`

规则：

1. `fire[i] = valid[i] && ready`
2. 首版 `ready` 对全 bundle 生效；后续可扩展为 `ready[i]`
3. 任意 lane 在 `valid[i]=1 && ready=0` 时必须保持 `bits[i]` 稳定

### 3.3 广播事件

`EventBus<E> { valid, event }`

适用场景：

1. flush/redirect
2. 中断注入
3. 分支解析广播（clear mask）

---

## 4. 前后端边界（强制）

边界必须固定为：

`FrontEnqueue -> UopBuffer -> Rename`

其中：

1. `FrontEnqueue` 只做协议驱动与节流。
2. `UopBuffer` 拥有自己的 ring 指针、满空判定、flush younger kill。
3. `Rename` 只消费 `UopBuffer` 输出，不能直连 `Decode`。

控制事件边界：

1. `be2fe_flush: Channel<FlushEvent>`
2. 可选 `BranchBroadcast`（仅 clear mask，不能替代全局 flush）

---

## 5. 统一数据结构契约（Schema v1）

### 5.1 `FrontUop`（前端语义层）

最小字段组：

1. 标识：`uop_id/seq_id/pc/isa_tag`
2. 运算：`op_class/sub_op`
3. 操作数：`src_en[]/src_areg[]/dst_en/dst_areg/imm`
4. 控制：`is_branch/is_call/is_ret/pred_taken/pred_target/br_id/br_mask`
5. 异常：`except_valid/except_code`
6. 验证：`is_microop/is_last_microop/is_fusion/is_mmio/is_sc/macro_id`
7. 访存/浮点属性：`mem_attr/fpr_writeback_mode`
8. 侧带：`dbg/tma`

### 5.2 `ExecUop`（后端执行层）

在 `FrontUop` 基础上新增：

1. `src_preg[]/dst_preg/old_dst_preg`
2. `rob_idx/rob_epoch`
3. `expect_mask/cplt_mask`
4. `fu_class/replay`

### 5.3 `CommitEntry`

最小字段组：

1. 提交标识：`valid/seq_id/pc`
2. 架构写回：`dst_en/dst_areg/dst_value`
3. 访存提交：`is_store/store_addr/store_data/store_mask`
4. 恢复提示：`except_valid/except_code/flush_pipe/mispred`
5. 侧带：`dbg/tma`

### 5.4 `FlushEvent`

最小字段组：

1. `valid`
2. `cause`（mispredict/exception/replay/fence）
3. `flush_seq_id`（年龄锚点）
4. `redirect_pc`
5. `rob_idx/rob_epoch`（可选锚点，便于硬件风格恢复）

---

## 6. 字段语义一致性规则

1. `seq_id`：全机唯一的程序顺序标识，不能在模块内重映射。
2. `rob_idx/rob_epoch`：仅 ROB owner 更新。
3. `br_mask`：只可被“分支解析 clear_mask”清位，禁止任意改写。
4. `except_code`：统一异常码空间，不能每级自定义编号。
5. `macro_id + is_last_microop`：用于宏指令边界聚合，必须由 Decode 保证自洽。

---

## 7. 并发语义（必须明确）

### 7.1 flush 与正常前推并发

1. flush 优先。
2. 本拍 younger 数据不应继续传播。

### 7.2 flush 与 backpressure 并发

1. 先执行 kill。
2. 再恢复握手。

### 7.3 多来源写回并发

1. 对同一目标条目必须定义固定仲裁。
2. `cplt_mask` 更新应为按位 OR，不得覆盖丢位。

---

## 8. 禁止事项

1. 禁止在边界结构中使用裸 `bool/int/uint32_t`（统一 `wire/reg`）。
2. 禁止模块绕过 IO 直接访问对方内部状态。
3. 禁止同一字段在不同模块定义不同语义。
4. 禁止引入“前端一套握手、后端另一套握手”。

---

## 9. 版本化建议

建议为协议引入 `io_schema_version`，规则如下：

1. 新增字段：小版本升级（向后兼容）。
2. 字段语义变化：大版本升级（需全链路对齐）。
3. 移除字段：仅在大版本进行，并提供迁移说明。
