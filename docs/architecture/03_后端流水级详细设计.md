# 后端流水级详细设计

## 1. 设计目标

后端必须满足三件事：

1. 只处理统一 `ExecUop`，不依赖 ISA 原始编码。
2. 按“乱序执行、顺序提交”保证架构态精确性。
3. 对 flush/exception/mispredict 提供可证明的恢复语义。

本章采用双层 uop 模型：

- `FrontUop`：前端语义 uop（ISA 翻译结果）。
- `ExecUop`：后端执行态（preg/rob/完成掩码/调度元数据）。

---

## 2. 状态所有权（先冻结，再实现）

### 2.1 Owner 表

1. `UopBuffer`
- 拥有 `head/tail/epoch`。
- 拥有前后端边界节流。

2. `Rename`
- 拥有 `specRAT/archRAT`。
- 拥有 `FreeList`。
- 拥有分支 checkpoint（RAT + alloc 快照）。

3. `Dispatch`
- 拥有分发路由与 IQ 端口占用决策。

4. `IssueQueue`
- 拥有队列条目、就绪位、选择仲裁状态。

5. `ROB`
- 拥有 `enq/deq + epoch`。
- 拥有提交顺序与恢复仲裁。

6. `LSQ`（后续）
- 拥有 `LDQ/STQ` 指针与顺序约束。

规则：跨模块只传值，不传可写指针引用。

---

## 3. 流水级定义

### 3.1 `Rename`

职责：
- 将 `src_areg/dst_areg` 映射为 `src_preg/dst_preg`。
- 分配新的目的 `dst_preg`，保留 `old_dst_preg`。
- 记录分支 checkpoint。

输入：
- `ChannelBundle<FrontUop, W>`。
- `Dispatch ready`、`FlushEvent`。

输出：
- `RenamedExecUopPacket`。

最小内部状态：
- `archRAT[AREG_NUM]`
- `specRAT[AREG_NUM]`
- `freeList[PRF_NUM]`
- `ratCheckpoint[BR_TAG_NUM][AREG_NUM]`

关键不变量：
1. `free + allocated == PRF_NUM`。
2. `archRAT` 只在 commit 更新。
3. `specRAT` 在 rename 前推更新，flush/mispredict 时回滚。

### 3.2 `Dispatch`

职责：
- 资源探测（ROB/IQ/LSQ 可用性）。
- 根据 `op_class/fu_class` 路由至不同 IQ。
- 为 `ExecUop` 填写 `expect_mask`。

输入：
- `RenamedExecUopPacket`
- 来自 IQ/ROB/LSQ 的 `ready` 信息

输出：
- `DisIssIO`、`DisRobIO`、`DisLsqIO`（后续）。

关键不变量：
1. 若任一核心资源不足，本拍不允许“部分语义提交”。
2. `expect_mask` 与功能分解一致。

### 3.3 `Issue`

职责：
- wakeup + ready 判定。
- 端口仲裁与发射。

输入：
- `DisIssIO`
- `wakeup` 广播
- `flush` 广播

输出：
- `IssExeIO`、`IssLsuIO`

关键不变量：
1. 同一条目同拍至多发射一次。
2. flush 时禁止新发射 younger。

### 3.4 `Exu`

职责：
- 非访存执行（INT/BR/CSR/FP 控制类）。
- 输出完成位来源（`cplt_mask` 子集）。

输出：
- `ExeWbIO`（含结果与异常/分支信息）。

### 3.5 `Lsu`

职责：
- 访存执行与次序控制。
- 支持 Load/Store/Atomic 事务语义。

输出：
- `LsuWbIO`
- `StoreCommitInfo`（后续可分离）。

### 3.6 `WriteBack`

职责：
- 汇合 Exu/Lsu 完成结果。
- 回写 PRF。
- 向 ROB 汇报完成来源位。
- 广播 wakeup。

关键约束：
- 同拍多来源冲突要有固定仲裁规则（端口优先级或 source-tag 优先级）。

### 3.7 `ROB + Commit`

职责：
- 维护程序序。
- 判定提交条件：`(cplt_mask & expect_mask) == expect_mask`。
- 生成 `CommitEntry` 与 `FlushEvent`。

关键约束：
1. 入队发生在 Dispatch（不是 WB）。
2. 完成位在 Complete 更新，不推进架构态。
3. 架构态仅在 Commit 更新。

---

## 4. ROB 生命周期（推荐标准）

每条 uop 生命周期应固定为：

1. `ALLOC`（Dispatch）
- 占 ROB 槽位，写入静态元数据（seq/pc/dst/except hint）。

2. `COMPLETE`（Exu/Lsu/WB）
- 仅更新 `cplt_mask` 与结果侧带。

3. `COMMIT`（ROB head）
- 满足完成条件后退休。
- 更新 `archRAT` 与架构寄存器可见状态。

4. `RECOVER/KILL`（Flush）
- kill younger。
- older 与已提交保留。

---

## 5. 指针与年龄模型

统一采用 `idx + epoch`。

### 5.1 满空判定

1. `idx==idx && epoch==epoch` -> empty
2. `idx==idx && epoch!=epoch` -> full

### 5.2 年龄比较

禁止直接比较裸整数 `idx` 或 `seq`。

必须使用统一接口：

1. `is_younger_seq(a, b, head_seq)`
2. `is_younger_ptr(a_idx, a_epoch, b_idx, b_epoch, head_idx, head_epoch)`

---

## 6. 完成掩码模型（多来源）

`ExecUop` 字段：

1. `expect_mask`
- 指令期望的完成源集合。

2. `cplt_mask`
- 已完成源集合。

提交条件：

`(cplt_mask & expect_mask) == expect_mask`

建议来源编码示例：

1. `bit0`: EXU 主结果
2. `bit1`: LSU load 数据
3. `bit2`: LSU store-addr/store-data 完成
4. `bit3`: CSR side effect 完成

---

## 7. 与前端协同

协同边界只保留两类信号：

1. 数据通道：`fe2be`（`FrontUop` bundle）
2. 控制事件：`FlushEvent`

禁止：
- 后端模块直接读前端内部状态。
- 前端直接读 ROB 内部数组。

---

## 8. 后端不变量（必须可断言）

1. 提交顺序与程序序一致。
2. younger 不得先于 older 提交。
3. flush 后 younger 全部失效。
4. `archRAT` 不被非 commit 路径修改。
5. `expect_mask/cplt_mask` 不出现越界位。
6. `ROB head` 若未完成，则提交停滞；完成后必须可前进（除非被 flush 抢占）。

---

## 9. 从当前骨架到目标架构的迁移路径

1. 第一步：把 ROB 入队点从 WB 挪到 Dispatch，先不改功能单元。
2. 第二步：引入 UopBuffer，解耦前后端背压。
3. 第三步：补齐 Rename 的 FreeList/checkpoint。
4. 第四步：把 Issue 从“直通”升级为真实 IQ + wakeup。
5. 第五步：补齐 LSQ 与原子事务语义。
