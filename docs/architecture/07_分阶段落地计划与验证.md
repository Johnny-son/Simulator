# 分阶段落地计划与验证

## 1. 阶段划分

### 阶段 A：协议冻结（先文档后代码）

目标：先冻结跨模块契约，避免实现过程中接口漂移。

完成标志：

1. 冻结 `IO/握手协议`（`Channel/Bundle/FlushEvent` 语义固定）。
2. 冻结 `FrontUop/ExecUop` 最小字段集合。
3. 冻结 `Flush/Redirect` 优先级与年龄比较规则。
4. 冻结多 ISA 前端插件接口（`DecoderRegistry`）。

### 阶段 B：单 ISA 闭环打通（RISC-V）

目标：在冻结协议下打通最小闭环。

路径：

`Decode -> FrontEnqueue -> UopBuffer -> Rename -> Dispatch -> Issue -> Exu/Lsu -> WriteBack -> ROB -> Commit`

完成标志：

1. 单发射/多发射基础路径可运行。
2. ROB 生命周期符合“Dispatch 入队、Complete 置位、Commit 退休”。
3. `expect_mask/cplt_mask` 提交条件可工作。

### 阶段 C：恢复语义闭环

目标：完成 mispredict/exception/replay/fence 的统一恢复。

完成标志：

1. flush younger kill 语义在前后端一致。
2. redirect 下一拍重启可复现。
3. flush 与背压/写回并发场景通过定向测试。

### 阶段 D：多 ISA 接入（x86 first, then N-ISA）

目标：接入第二 ISA 且不改后端主路径。

完成标志：

1. x86 解码器通过 `DecoderRegistry` 注册。
2. 后端不新增 ISA 分支判断。
3. 双 ISA 共用同一 `FrontUop/ExecUop/FlushEvent` 协议。

### 阶段 E：验证与性能增强

目标：在协议稳定基础上做验证和性能优化。

完成标志：

1. 模块级断言与 trace 回放齐备。
2. 指令级 difftest（DUT/REF）可运行。
3. 核心性能统计（front/back bound, flush cost）可观测。

---

## 2. 最小验证用例

1. 直线指令流（无分支）。
2. 分支命中预测。
3. 分支错预测回滚。
4. 异常触发并重定向。
5. 背压场景（`ready=0` 连续若干拍）。
6. 同拍 `mispredict + exception` 仲裁。
7. 指针回卷边界（head/tail/seq 跨界）。
8. flush 与写回/背压并发。
9. 双 ISA 混合输入（先用定向 micro trace）。

---

## 3. 质量门禁建议

1. Build 门禁
- 每次协议字段修改后全量编译。

2. 接口门禁
- `FrontUop/ExecUop/FlushEvent` 结构变化必须更新文档版本号。

3. 语义门禁
- flush 相关改动必须附带并发测试（至少 3 个冲突用例）。

4. 验证门禁
- 至少保留 1 个前后端闭环 smoke test + 1 个恢复语义 regression。

---

## 4. 迁移顺序建议（对齐启蒙3号）

1. 先借鉴启蒙3号的“组合逻辑分组顺序”和“ROB 生命周期约束”。
2. 再引入 `UopBuffer`，消除前后端硬直连。
3. 然后补齐 Rename（FreeList/checkpoint）与 Dispatch（expect_mask）。
4. 最后接入 x86 解码插件，不改后端协议。
