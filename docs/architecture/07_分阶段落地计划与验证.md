# 分阶段落地计划与验证

## 1. 阶段划分

### 阶段 A：底层类型基线（wire/reg）

目标：建立统一 `wire<N>/reg<N>` 类型系统，成为前后端唯一边界类型。

完成标志：

1. 核心头文件存在并可独立编译。
2. `IO` 与 `uop` 关键字段不再使用裸整型。
3. 关键位宽有 `static_assert`。

### 阶段 B：协议冻结与单路径打通

目标：冻结 `FrontUop/ExecUop`、QM3 同名级间 IO（`DecRenIO`、`RenDisIO`、`DisIssIO` 等）、`FlushEvent` 并打通 `Decode -> FrontEnqueue -> Rename -> Dispatch -> Issue -> (Exu|Lsu) -> WriteBack -> Commit`。

完成标志：单发射下可跑完整生命周期。

### 阶段 C：指针体系与恢复语义接入

目标：在 ROB/UopBuffer/IQ/LSQ 接入 `idx+epoch` 指针体系与统一年龄比较，完成错预测与异常恢复闭环。

完成标志：回滚后可从 `redirect_pc` 正常继续执行。

### 阶段 D：扩展与性能

目标：多发射、更多功能单元、统计指标与调优。

## 2. 最小验证用例

1. 正常直线指令流（无分支）
2. 分支命中预测
3. 分支错预测回滚
4. 异常触发并重定向
5. 背压场景（`ready=0` 连续若干拍）
6. 同拍 `mispredict + exception` 仲裁
7. 指针回卷边界（head/tail/seq 跨界）
8. flush 与写回/背压并发

## 3. 质量门禁建议

- Build：每次接口修改后全量编译
- Lint/Typecheck：保持头文件契约一致
- Test：至少保留一个前后端闭环 smoke test

## 4. 迁移顺序建议（从启蒙3号搬运）

1. 先搬 `base_types`（`wire/reg/WideBits`）并在新项目编译通过。
2. 再搬 QM3 同名 `IO` 契约（不搬具体模块实现），让前后端对齐同一协议。
3. 之后逐步搬 `ROB/Rename/Issue` 的指针与恢复机制。
4. 最后搬性能优化与统计，不在早期阶段引入复杂策略。
